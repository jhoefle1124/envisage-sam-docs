Pillar 6 · Delivery Layer

MCP Output and Integration

How SAM-AI's tools, data, and analysis reach the user — exposed through the Model Context Protocol so a Claude Cowork-style workflow can call them, chain them into multi-step work, and hand back finished files, live views, and scheduled reports.

Why This Layer Exists

The first five pillars build SAM-AI's intelligence: a model, directed by firm instructions, reasoning over firm and team data, with its output scored and tracked. None of that is useful until it reaches someone in a form they can actually act on. MCP Output and Integration is the interface layer that makes that handoff possible — it takes the internal functions, data access, and agent coordination described in the FM Configured AI Instructions pillar and exposes them through the Model Context Protocol (MCP), the open standard that lets an agentic client like Claude Cowork discover what SAM-AI can do, call it directly, and turn the result into a delivered piece of work rather than a chat transcript.

Where Pillar 2 defines what the AI can reason about and act on, this pillar defines how that capability gets packaged and handed to a workflow — as a callable tool, a browsable resource, a generated file, or a running scheduled task.

MCP Server Architecture

The plumbing that turns SAM-AI's internal functions into something an external agent can discover and call.

Tool Exposure

SAM-AI's business logic, database queries, and valuation functions are registered as MCP tools with defined input and output schemas, so an agent can call a scorecard calculation or a data lookup the same way it would call any other function — without needing to know how it's implemented underneath.

Resource & Context Exposure

SAM Data and SAM Team Data are surfaced as browsable MCP resources — documents, records, and embeddings the connected client can read directly — rather than requiring every access to go through a bespoke tool call.

Deferred Tool Loading

Because a fully integrated SAM-AI instance can expose dozens of tools across data, analysis, and delivery, schemas are loaded on demand rather than all at once — the agent sees a tool's name first and only pulls its full definition when the task actually calls for it, keeping large toolsets fast and manageable.

Connector Integrations

The channels through which a Cowork-style workflow reaches the systems the team already works in.

File & Folder Connectors

Scoped access to a specific working folder — local or cloud-based — so the AI can read source material and write finished deliverables directly where the team will find them, without broad access to an entire drive.

Productivity & Data Connectors

Integrations with the tools a research or portfolio team already uses — messaging, project tracking, document platforms, and the firm's own databases — so SAM-AI's output lands inside the existing workflow instead of a separate silo.

Authentication & Scoped Access

Every connector is authorized explicitly by the user before use, through the credential flow appropriate to that service, and is limited to the narrowest scope the task requires.

Cowork Workflow Patterns

How MCP tool calls are sequenced into an actual piece of multi-step work, rather than a single request and response.

Task Planning & Progress Tracking

Multi-step work is broken into a visible task list before execution begins, so the user can see what the AI intends to do and track progress as each step completes.

Multi-step Agentic Execution

Tool calls are chained together — querying SAM Data, running a valuation scorecard, and drafting a deliverable — with each step's output feeding the next, so a single request can complete an entire piece of research rather than one lookup at a time.

Human-in-the-Loop Checkpoints

Before committing to an ambiguous course of action, or before a high-stakes step, the workflow pauses to ask the user a clarifying question rather than guessing — keeping a person in control of any consequential decision.

Output Delivery

The forms SAM-AI's finished work actually takes once a workflow completes.

File Artifacts

Analysis and research are delivered as finished documents, spreadsheets, decks, or PDFs — formatted, saved to the user's workspace, and ready to open, rather than left as raw text in a chat window.

Live Artifacts

Recurring views — a portfolio dashboard, a status tracker, a comparative scorecard — are delivered as persistent pages that pull fresh data from connectors each time they're reopened, instead of a one-time snapshot.

Scheduled Tasks

Work that needs to happen on a cadence — a daily market brief, a weekly performance summary — is set up to run automatically, so the output is waiting for the team rather than requiring a fresh request each time.

Governance & Guardrails

The same discipline that governs SAM Data access applies to every MCP tool call and every delivered output.

Minimum Necessary Access

A workflow gets the one connector or folder its task requires, not standing access to everything the user could theoretically reach — narrowing the blast radius of any single request.

Sandboxed Execution

Any code or script execution a tool call triggers runs in an isolated environment, separate from production systems and the user's own machine.

Audit Logging

Every tool call, connector access, and delivered artifact is recorded, feeding directly into the decision tracking described under SAM Analysis & Decision Monitoring — so an AI-assisted deliverable is traceable back to exactly which tools and data produced it.

How it connects back: MCP Output and Integration doesn't add new intelligence to SAM-AI — it exposes the intelligence built in Pillars 1 through 4, and feeds the record of what it delivered into the accountability loop closed by Pillar 5.
