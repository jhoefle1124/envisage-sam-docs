Section 4

Cross-Verifying with Gemini & ChatGPT

Every model has blind spots, and our deliverables carry our name to UN agencies, funds, and corporate clients. A second model is a cheap independent check — when used on the right things, the right way.

The Principle

Cross-verification means taking a specific claim or output produced with one model and independently checking it with another (Gemini or ChatGPT). It is not re-doing the whole task twice. Two rules govern every verification:

1

Verify the claim, not the conversation

Paste only the sentence, figure, or code block being checked — never the source document, the chat history, or surrounding analysis. This protects confidentiality and costs a fraction of the tokens.

2

Tier 2 maximum, sanitized

Verification tools never receive Tier 3 content. Anonymize first (see Data Security). If a claim can't be checked without revealing restricted detail, verify it against primary sources by hand instead.

Agreement is not truth. Two models can share the same wrong answer, especially on recent events or thin-data regions like those we cover. Model agreement raises confidence; only a primary source settles a fact destined for a client deliverable.

When to Verify — and With What

SituationVerify?How
Statistic or factual claim going into a client deliverableAlwaysAsk Gemini or ChatGPT to confirm/refute with sources; then check the primary source yourself.
Translation of source material (Burmese, Thai, Shan, etc.) affecting a findingAlwaysSecond model translates the same sanitized passage blind; compare meanings. Disagreement → human translator.
Legal, regulatory, or sanctions-related statementsAlwaysCross-check, then confirm against the official text. Never deliver on model output alone.
SQL or Python destined for the Sapphire pipelineUsuallyPaste the code (schema names only, no data) into a second model: "Find bugs and edge cases." Then test on dev data.
Summary of a public documentSpot-checkVerify the 2–3 claims you'd be most embarrassed to get wrong.
Internal brainstorm, draft wording, formattingNoEditing judgment, not facts. Verification here is token waste.

Worked Examples

Example 1 — Statistic in a deliverable

Scenario

Claude drafts: "The top ten industrialized nations spend a combined ~$200 billion annually on development and foreign aid." This is headed for a client report.

To Gemini: "What is the most recent combined ODA spending of the top ten donor nations? Cite the source." If Gemini's figure differs materially, pull the OECD data directly and use the primary number with a citation. Cost: ~100 tokens. Value: our credibility.

Example 2 — Translation check

Scenario

A harvested Burmese-language article appears to say a company "acquired" a concession; the finding hinges on whether it means acquired or applied for.

To ChatGPT (sanitized passage, names replaced): "Translate this passage precisely. Does the verb indicate completed acquisition or a pending application?" If the models disagree, the passage goes to a human translator on the team before the finding is written.

Example 3 — SQL for the pipeline

Scenario

Claude wrote a Snowflake query joining three tables to flag entities appearing across datasets.

To Gemini: paste the query plus a one-line schema description: "Review for logic errors, join fan-out, and NULL handling." Models are excellent at catching each other's off-by-one logic. Then run on dev data before production.

Example 4 — When NOT to verify

Scenario

Claude restructured your interview notes into themes. There is no external fact to check — review it yourself. Sending it to a second model adds token cost and a second exposure point for zero gain.

Verification Etiquette (Token-Efficient)

  • Blind the second model. Don't say "Claude said X — is that right?" Ask the question fresh. Models anchor on suggested answers.
  • Batch your checks. Collect the deliverable's 5 key claims and verify them in one message.
  • Use free tiers for verification. Verification prompts are short and contextless — Gemini or ChatGPT's free tier is almost always sufficient; save Claude capacity for work that needs our context.
  • Stop at disagreement, not at agreement. If the second model confirms Claude's answer, move on. If it disagrees, that's the signal to check the primary source yourself.

การตรวจสอบไขว้กับ Gemini และ ChatGPT

โมเดลทุกตัวมีจุดบอด และงานส่งมอบของเรานำชื่อของเราไปสู่หน่วยงานสหประชาชาติ กองทุน และลูกค้าองค์กร โมเดลที่สองคือการตรวจสอบอิสระที่มีต้นทุนต่ำ — เมื่อใช้กับสิ่งที่ถูกต้อง ด้วยวิธีที่ถูกต้อง

หลักการ

การตรวจสอบไขว้ (Cross-verification) หมายถึงการนำข้อความอ้างหรือผลลัพธ์ที่เจาะจงซึ่งสร้างขึ้นด้วยโมเดลหนึ่ง มาตรวจสอบอย่างอิสระด้วยอีกโมเดลหนึ่ง (Gemini หรือ ChatGPT) ไม่ใช่การทำงานทั้งหมดซ้ำสองครั้ง กฎสองข้อควบคุมการตรวจสอบทุกครั้ง:

1

ตรวจสอบข้อความอ้าง ไม่ใช่บทสนทนาทั้งหมด

วางเฉพาะประโยค ตัวเลข หรือ code block ที่กำลังตรวจสอบเท่านั้น — ห้ามส่งเอกสารต้นฉบับ ประวัติการสนทนา หรือการวิเคราะห์โดยรอบ วิธีนี้ช่วยปกป้องความลับและใช้ token เพียงเสี้ยวเดียว

2

สูงสุดที่ Tier 2 และต้อง sanitize แล้ว

เครื่องมือตรวจสอบต้องไม่ได้รับเนื้อหา Tier 3 เด็ดขาด ให้ทำให้ไม่ระบุตัวตนก่อน (ดู Data Security) หากข้อความอ้างไม่สามารถตรวจสอบได้โดยไม่เปิดเผยรายละเอียดที่ถูกจำกัด ให้ตรวจสอบกับแหล่งข้อมูลปฐมภูมิด้วยตนเองแทน

ความเห็นตรงกันไม่ใช่ความจริง โมเดลสองตัวอาจให้คำตอบผิดเหมือนกันได้ โดยเฉพาะเหตุการณ์ล่าสุดหรือพื้นที่ที่มีข้อมูลน้อยอย่างที่เราครอบคลุม ความเห็นตรงกันของโมเดลช่วยเพิ่มความมั่นใจ แต่มีเพียงแหล่งข้อมูลปฐมภูมิเท่านั้นที่ยืนยันข้อเท็จจริงซึ่งจะนำไปใช้ใน client deliverable ได้

เมื่อใดควรตรวจสอบ — และด้วยอะไร

สถานการณ์ต้องตรวจสอบหรือไม่วิธีการ
สถิติหรือข้อความอ้างเชิงข้อเท็จจริงที่จะนำไปใช้ใน client deliverableเสมอให้ Gemini หรือ ChatGPT ยืนยันหรือหักล้างพร้อมแหล่งอ้างอิง จากนั้นตรวจสอบแหล่งข้อมูลปฐมภูมิด้วยตนเอง
การแปลเอกสารต้นฉบับ (พม่า ไทย ไทใหญ่ ฯลฯ) ที่ส่งผลต่อข้อค้นพบเสมอให้โมเดลที่สองแปลข้อความที่ sanitize แล้วชุดเดียวกันแบบ blind แล้วเปรียบเทียบความหมาย หากไม่ตรงกัน → ส่งให้นักแปลที่เป็นมนุษย์
ข้อความด้านกฎหมาย ระเบียบข้อบังคับ หรือ sanctionsเสมอตรวจสอบไขว้ แล้วยืนยันกับข้อความทางการ ห้ามส่งมอบงานโดยอ้างอิงผลลัพธ์จากโมเดลเพียงอย่างเดียว
SQL หรือ Python ที่จะใช้ใน Sapphire pipelineโดยปกติวางโค้ด (เฉพาะชื่อ schema ไม่มีข้อมูลจริง) ลงในโมเดลที่สอง: "หาบั๊กและ edge case" จากนั้นทดสอบกับ dev data
การสรุปเอกสารสาธารณะสุ่มตรวจตรวจสอบข้อความอ้าง 2–3 ข้อที่หากผิดแล้วจะน่าอายที่สุด
การระดมความคิดภายใน ร่างถ้อยคำ การจัดรูปแบบไม่ต้องเป็นเรื่องดุลยพินิจในการเขียน ไม่ใช่ข้อเท็จจริง การตรวจสอบตรงนี้เป็นการเสีย token โดยเปล่าประโยชน์

ตัวอย่างการใช้งานจริง

ตัวอย่างที่ 1 — สถิติใน deliverable

สถานการณ์

Claude ร่างข้อความว่า: "ประเทศอุตสาหกรรมสิบอันดับแรกใช้จ่ายด้านการพัฒนาและความช่วยเหลือต่างประเทศรวมกันประมาณ 2 แสนล้านดอลลาร์ต่อปี" ข้อความนี้กำลังจะถูกนำไปใช้ในรายงานลูกค้า

ถามไปที่ Gemini: "ตัวเลขการใช้จ่าย ODA รวมล่าสุดของประเทศผู้บริจาคสิบอันดับแรกคือเท่าไร กรุณาอ้างอิงแหล่งที่มา" หากตัวเลขของ Gemini แตกต่างอย่างมีนัยสำคัญ ให้ดึงข้อมูล OECD โดยตรงและใช้ตัวเลขจากแหล่งปฐมภูมิพร้อมอ้างอิง ต้นทุน: ประมาณ 100 token คุณค่า: ความน่าเชื่อถือของเรา

ตัวอย่างที่ 2 — การตรวจสอบการแปล

สถานการณ์

บทความภาษาพม่าที่เก็บรวบรวมมาดูเหมือนจะระบุว่าบริษัทหนึ่ง "acquired" สัมปทาน ข้อค้นพบขึ้นอยู่กับว่าคำนี้หมายถึงได้มาแล้วหรือเพียงยื่นขอ

ถามไปที่ ChatGPT (ข้อความที่ sanitize แล้ว เปลี่ยนชื่อทั้งหมด): "แปลข้อความนี้อย่างละเอียด คำกริยานี้หมายถึงการได้มาซึ่งเสร็จสมบูรณ์แล้ว หรือคำร้องที่ยังรอดำเนินการ?" หากโมเดลให้คำตอบไม่ตรงกัน ให้ส่งข้อความนี้ไปยังนักแปลที่เป็นมนุษย์ในทีมก่อนเขียนข้อค้นพบ

ตัวอย่างที่ 3 — SQL สำหรับ pipeline

สถานการณ์

Claude เขียนคำสั่ง Snowflake query ที่ join สามตารางเพื่อตรวจหา entity ที่ปรากฏในหลายชุดข้อมูล

ถามไปที่ Gemini: วางคำสั่ง query พร้อมคำอธิบาย schema สั้น ๆ หนึ่งบรรทัด: "ตรวจสอบข้อผิดพลาดเชิงตรรกะ join fan-out และการจัดการ NULL" โมเดลมีความสามารถดีเยี่ยมในการตรวจจับข้อผิดพลาดเชิงตรรกะแบบ off-by-one ของกันและกัน จากนั้นรันบน dev data ก่อนนำขึ้น production

ตัวอย่างที่ 4 — กรณีที่ไม่ควรตรวจสอบ

สถานการณ์

Claude จัดโครงสร้างบันทึกการสัมภาษณ์ของคุณใหม่ให้เป็นหมวดหมู่ตามธีม ไม่มีข้อเท็จจริงภายนอกที่ต้องตรวจสอบ — ให้ทบทวนด้วยตนเอง การส่งไปยังโมเดลที่สองมีแต่จะเพิ่มต้นทุน token และจุดเสี่ยงที่สองโดยไม่ได้ประโยชน์ใด ๆ

มารยาทการตรวจสอบ (ประหยัด Token)

  • ปิดบังข้อมูลจากโมเดลที่สอง (Blind) อย่าพูดว่า "Claude บอกว่า X — ถูกไหม" ให้ถามคำถามใหม่โดยไม่ชี้นำ โมเดลมักจะยึดติดกับคำตอบที่ถูกเสนอแนะไว้ก่อน
  • รวมการตรวจสอบเป็นชุด รวบรวมข้อความอ้างสำคัญ 5 ข้อของ deliverable แล้วตรวจสอบในข้อความเดียว
  • ใช้ free tier สำหรับการตรวจสอบ prompt สำหรับตรวจสอบสั้นและไม่มีบริบทมาก — free tier ของ Gemini หรือ ChatGPT มักเพียงพอเสมอ เก็บโควตาของ Claude ไว้ใช้กับงานที่ต้องการบริบทของเรา
  • หยุดเมื่อไม่ตรงกัน ไม่ใช่เมื่อเห็นตรงกัน หากโมเดลที่สองยืนยันคำตอบของ Claude ให้ดำเนินการต่อ หากไม่ตรงกัน นั่นคือสัญญาณให้ตรวจสอบแหล่งข้อมูลปฐมภูมิด้วยตนเอง

အပိုင်း ၄

Gemini နှင့် ChatGPT ဖြင့် ပြန်လည်စစ်ဆေးခြင်း (Cross-Verification)

မော်ဒယ်တိုင်းတွင် မမြင်နိုင်သော အားနည်းချက်များရှိပြီး ကျွန်ုပ်တို့၏ deliverable များသည် ကျွန်ုပ်တို့၏အမည်ကို UN အေဂျင်စီများ၊ ရန်ပုံငွေများ နှင့် ကော်ပိုရိတ်ဖောက်သည်များထံ ယူဆောင်သွားသည်။ ဒုတိယမော်ဒယ်သည် — မှန်ကန်သောအရာများကို မှန်ကန်သောနည်းလမ်းဖြင့် သုံးမှသာ — ကုန်ကျစရိတ်နည်းသော လွတ်လပ်သည့် စစ်ဆေးမှု တစ်ခုဖြစ်သည်။

အခြေခံမူ

Cross-verification ဆိုသည်မှာ မော်ဒယ်တစ်ခုဖြင့် ထုတ်လုပ်ထားသော တိကျသော claim သို့မဟုတ် ရလဒ် တစ်ခုကို အခြားမော်ဒယ် (Gemini သို့မဟုတ် ChatGPT) ဖြင့် သီးခြားစွာ စစ်ဆေးခြင်းကို ဆိုလိုသည်။ တာဝန်တစ်ခုလုံးကို နှစ်ကြိမ်ပြန်လုပ်ခြင်း မဟုတ်ပါ။ စစ်ဆေးမှုတိုင်းကို စည်းမျဉ်းနှစ်ချက် ထိန်းချုပ်သည်-

1

claim ကိုသာ စစ်ဆေးပါ၊ စကားဝိုင်းတစ်ခုလုံးကို မဟုတ်

စစ်ဆေးနေသော ဝါကျ၊ ကိန်းဂဏန်း သို့မဟုတ် code block ကိုသာ ကူးထည့်ပါ — မူရင်းစာတမ်း၊ chat မှတ်တမ်း သို့မဟုတ် ပတ်ဝန်းကျင် analysis ကို ဘယ်တော့မှ မထည့်ရ။ ၎င်းသည် လျှို့ဝှက်ချက်ကို ကာကွယ်ပေးပြီး token ၏ အစိတ်အပိုင်းငယ်တစ်ခုသာ ကုန်ကျစေသည်။

2

အများဆုံး Tier 2၊ sanitize ပြီးသား

စစ်ဆေးရေးကိရိယာများသို့ Tier 3 အကြောင်းအရာ ဘယ်တော့မှ မပေးရ။ ဦးစွာ အမည်ဖျောက်ပါ (Data Security ကိုကြည့်ပါ)။ claim တစ်ခုကို ကန့်သတ်အသေးစိတ် မဖော်ပြဘဲ စစ်ဆေး၍မရပါက ၎င်းအစား primary source များနှင့် ကိုယ်တိုင်စစ်ဆေးပါ။

သဘောတူညီမှုသည် အမှန်တရားမဟုတ်ပါ။ မော်ဒယ်နှစ်ခုသည် အထူးသဖြင့် လတ်တလောဖြစ်ရပ်များ သို့မဟုတ် ကျွန်ုပ်တို့ လွှမ်းခြုံသည့် ဒေတာနည်းသောနေရာများတွင် တူညီသောအမှားကို မျှဝေနိုင်သည်။ မော်ဒယ်များ သဘောတူညီမှုသည် ယုံကြည်မှုကို မြှင့်တင်ပေးသော်လည်း၊ ဖောက်သည် deliverable တွင် ထည့်မည့် အချက်အလက်ကို primary source တစ်ခုတည်းကသာ အတည်ပြုနိုင်သည်။

ဘယ်အချိန်တွင် ဘာဖြင့် စစ်ဆေးမည်နည်း

အခြေအနေစစ်ဆေးရမလားနည်းလမ်း
ဖောက်သည် deliverable တွင် ထည့်မည့် စာရင်းအင်း သို့မဟုတ် အချက်အလက်ဆိုင်ရာ claimအမြဲစစ်ဆေးရမည်Gemini သို့မဟုတ် ChatGPT ကို source များနှင့်တကွ အတည်ပြု/ငြင်းချေခိုင်းပါ၊ ထို့နောက် primary source ကို ကိုယ်တိုင်စစ်ဆေးပါ
တွေ့ရှိချက်တစ်ခုကို ထိခိုက်စေသော မူရင်းရင်းမြစ် ဘာသာပြန်ချက် (မြန်မာ၊ ထိုင်း၊ ရှမ်း စသည်)အမြဲစစ်ဆေးရမည်ဒုတိယမော်ဒယ်ကို sanitize ပြီးသား စာပိုဒ်တူကို blind ဘာသာပြန်စေပါ၊ အဓိပ္ပာယ်များကို နှိုင်းယှဉ်ပါ။ သဘောထားမကွဲလွဲပါက → လူ ဘာသာပြန်သူထံ ပေးပို့ပါ
ဥပဒေရေးရာ၊ စည်းမျဉ်းဆိုင်ရာ သို့မဟုတ် ပိတ်ဆို့မှုနှင့်ဆက်စပ်သော ဖော်ပြချက်များအမြဲစစ်ဆေးရမည်Cross-check လုပ်ပြီး တရားဝင် စာသားနှင့် အတည်ပြုပါ။ မော်ဒယ်ရလဒ်တစ်ခုတည်းအပေါ် အခြေခံ၍ ဘယ်တော့မှ ပေးအပ်ခြင်း မပြုရ
Sapphire pipeline သို့ ရည်ရွယ်သော SQL သို့မဟုတ် Pythonများသောအားဖြင့်code ကို (schema အမည်များသာ၊ ဒေတာမပါ) ဒုတိယမော်ဒယ်တွင် ကူးထည့်ပါ- "bug များနှင့် edge case များ ရှာပါ" ထို့နောက် dev data ဖြင့် စမ်းသပ်ပါ
အများပြည်သူစာတမ်း၏ အကျဉ်းချုပ်သုံးသပ်စစ်ဆေးပါအမှားရှိလျှင် အရှက်အကြီးဆုံးဖြစ်စေမည့် claim ၂-၃ ခုကို စစ်ဆေးပါ
အတွင်းပိုင်း idea ထုတ်ခြင်း၊ ရေးသားချက်မူကြမ်း၊ ပုံစံချခြင်းမလိုအပ်ပါဤသည်မှာ တည်းဖြတ်ရေး ဆုံးဖြတ်ချက်ဖြစ်ပြီး အချက်အလက်မဟုတ်ပါ။ ဤနေရာတွင် စစ်ဆေးခြင်းသည် token ဖြုန်းတီးမှုသာ ဖြစ်သည်

လက်တွေ့ ဥပမာများ

ဥပမာ ၁ — Deliverable ထဲက စာရင်းအင်း

အခြေအနေ

Claude က ဤသို့ မူကြမ်းရေးသည်- "စက်မှုထွန်းကားသော ထိပ်တန်း နိုင်ငံ ၁၀ နိုင်ငံသည် ဖွံ့ဖြိုးရေးနှင့် နိုင်ငံခြားအကူအညီအတွက် တစ်နှစ်လျှင် ခန့်မှန်းခြေ အမေရိကန်ဒေါ်လာ ၂၀၀ ဘီလီယံ ပေါင်းစပ်၍ သုံးစွဲသည်။" ဤအချက်သည် ဖောက်သည်အစီရင်ခံစာသို့ သွားနေသည်။

Gemini ကို မေးရန်: "ထိပ်တန်း အလှူရှင်နိုင်ငံ ၁၀ နိုင်ငံ၏ နောက်ဆုံး ပေါင်းစည်း ODA သုံးစွဲမှု ဘယ်လောက်ရှိသလဲ။ source ကို ကိုးကားပါ" Gemini ၏ ကိန်းဂဏန်းသည် သိသိသာသာ ကွာခြားပါက OECD ဒေတာကို တိုက်ရိုက်ရယူပြီး primary ကိန်းဂဏန်းကို ကိုးကားချက်နှင့်တကွ သုံးပါ။ ကုန်ကျစရိတ်- token ~၁၀၀ ခန့်။ တန်ဖိုး- ကျွန်ုပ်တို့၏ ယုံကြည်စိတ်ချရမှု။

ဥပမာ ၂ — ဘာသာပြန်ချက် စစ်ဆေးခြင်း

အခြေအနေ

ရယူထားသော မြန်မာဘာသာ ဆောင်းပါးတစ်ခုက ကုမ္ပဏီတစ်ခုသည် သတ္တုတွင်းလုပ်ပိုင်ခွင့်ကို "acquired" ရရှိသည်ဟု ဆိုနေပုံရသည်။ တွေ့ရှိချက်သည် ၎င်းက လုံးဝ ရရှိပြီးလား သို့မဟုတ် လျှောက်ထားဆဲလားဆိုသည်ပေါ် မူတည်နေသည်။

ChatGPT ကို (sanitize ပြီးသား စာပိုဒ်၊ အမည်များ အစားထိုးထား): "ဤစာပိုဒ်ကို တိကျစွာ ဘာသာပြန်ပါ။ ဤကြိယာသည် ပြီးစီးသော ရယူမှုကို ဆိုလိုသလား၊ သို့မဟုတ် ဆိုင်းငံ့နေသေးသော လျှောက်လွှာကို ဆိုလိုသလား?" မော်ဒယ်များ သဘောထားကွဲလွဲပါက စာပိုဒ်ကို တွေ့ရှိချက် မရေးမီ အဖွဲ့ရှိ လူ ဘာသာပြန်သူထံ ပေးပို့ပါ။

ဥပမာ ၃ — Pipeline အတွက် SQL

အခြေအနေ

Claude သည် ဒေတာစုများစွာတွင် ပေါ်နေသော entity များကို ဖော်ထုတ်ရန် ဇယားသုံးခု join လုပ်သော Snowflake query တစ်ခု ရေးခဲ့သည်။

Gemini ကို: query နှင့်အတူ schema ဖော်ပြချက် တစ်ကြောင်းတည်း ကူးထည့်ပါ- "logic error, join fan-out နှင့် NULL handling ကို စစ်ဆေးပါ" မော်ဒယ်များသည် တစ်ခုနှင့်တစ်ခု၏ off-by-one logic အမှားများကို ဖော်ထုတ်ရာတွင် အလွန်ကောင်းမွန်သည်။ ထို့နောက် production မတင်မီ dev data ဖြင့် run ပါ။

ဥပမာ ၄ — မစစ်ဆေးသင့်သည့် အချိန်

အခြေအနေ

Claude သည် သင့် interview မှတ်စုများကို theme အလိုက် ပြန်လည်ဖွဲ့စည်းပေးခဲ့သည်။ စစ်ဆေးရန် ပြင်ပ အချက်အလက် မရှိပါ — ကိုယ်တိုင် ပြန်ကြည့်ပါ။ ဒုတိယမော်ဒယ်ထံ ပို့ခြင်းသည် အကျိုးမရှိဘဲ token ကုန်ကျစရိတ်နှင့် ဒုတိယ ထိတွေ့မှု အမှတ်တစ်ခု ထပ်တိုးစေရုံသာ ဖြစ်သည်။

စစ်ဆေးရေး ကျင့်ဝတ် (Token ချွေတာသော)

  • ဒုတိယမော်ဒယ်ကို Blind ထားပါ။ "Claude က X လို့ပြောတယ် — မှန်လား" ဟူ၍ မမေးပါနှင့်။ မေးခွန်းကို အသစ်မေးပါ။ မော်ဒယ်များသည် အကြံပြုထားသော အဖြေများကို ကိုးကားတတ်သည်။
  • စစ်ဆေးမှုများကို အတွဲလိုက် လုပ်ပါ။ Deliverable ၏ အဓိက claim ၅ ခုကို စုပြီး message တစ်ခုတည်းတွင် စစ်ဆေးပါ။
  • စစ်ဆေးရေးအတွက် free tier ကို သုံးပါ။ စစ်ဆေးရေး prompt များသည် တိုတောင်း၍ context မလိုအပ်ပါ — Gemini သို့မဟုတ် ChatGPT ၏ free tier သည် အများအားဖြင့် လုံလောက်သည်။ Claude ၏ စွမ်းရည်ကို ကျွန်ုပ်တို့ context လိုအပ်သော အလုပ်များအတွက် သိမ်းထားပါ။
  • သဘောထားမကွဲလွဲသည့်နေရာတွင် ရပ်ပါ၊ သဘောတူသည့်နေရာတွင် မဟုတ်ပါ။ ဒုတိယမော်ဒယ်က Claude ၏ အဖြေကို အတည်ပြုပါက ဆက်သွားပါ။ သဘောထားကွဲလွဲပါက ၎င်းသည် primary source ကို ကိုယ်တိုင် စစ်ဆေးရန် အချက်ပြမှု ဖြစ်သည်။
