Section 5

The Wider AI Toolkit

Free public LLMs and specialized coding agents let us reserve paid Claude capacity for the work that needs it. The trade-off is control: the cheaper the tool, the less sensitive the content it may touch.

Free Public LLMs: DeepSeek & Kimi

DeepSeek (DeepSeek AI) and Kimi (Moonshot AI) are capable free models — strong at coding, long-document handling, and general drafting, often rivaling paid models on generic tasks. They are excellent token-savers for Tier 1 work.

Hard limit: Tier 1 only. These services are hosted outside our control, with data-handling terms we cannot enforce — and given where ENVISAGE operates and what we research, our subject matter is uniquely sensitive in that context. Nothing internal, nothing client-related, no names from our research — ever. Assume everything sent is retained and readable.

Good uses (Tier 1)

  • Summarizing published news articles and public reports during open-source harvesting triage
  • First-pass generic drafting: boilerplate descriptions, meeting agendas, generic outreach templates
  • Explaining a public concept, statistical method, or library before you spend Claude tokens applying it to our context
  • Generic code snippets and debugging help (no schema names, no internal code)
  • Brainstorming structures and checklists with no ENVISAGE specifics
Good routing

An analyst triaging 30 public articles asks Kimi for two-sentence summaries of each, picks the 6 relevant ones, then brings only those into Claude for contextual analysis against the engagement. Free tool does the bulk reading; Claude does the judgment.

Never

Pasting a draft client assessment into DeepSeek "just to tighten the wording." The wording is attached to Tier 3 content — that draft now lives on infrastructure we don't control.

Coding Agents: Plain Language to SQL and Beyond

Natural-language-to-SQL agents (Pi-style NL2SQL tools, SQL Agent CLI, and similar) and coding assistants turn plain-English requests into working queries and scripts. For a team whose stack runs on Python, PostgreSQL, Elasticsearch, and Snowflake, they let analysts who don't write SQL get answers without queuing for a developer.

The safe pattern: schema in, query out

A coding agent never needs your data to write your query — it needs the shape of your data. Describe tables and columns; run the generated query yourself inside our environment.

Our PostgreSQL database has:
  entities(entity_id, name_hash, type, country, first_seen)
  relationships(rel_id, entity_a, entity_b, rel_type, confidence)
  mentions(mention_id, entity_id, source_id, published_date)

Write a query returning entities mentioned in 3+ sources
in the last 90 days that have at least one 'ownership'
relationship, ordered by mention count.

Note what the prompt contains: column names and intent — no rows, no real names, no credentials. The agent returns SQL; you review it, cross-check it if it's pipeline-bound, and execute it inside Sapphire.

Rules for coding agents

  • Schema structure only. Table and column names are Tier 2 — fine for approved tools. Data rows are whatever tier the data is, and usually Tier 3.
  • Never paste credentials or connection strings, and never let an external agent connect directly to production databases. Generated queries run through our own access controls.
  • Test on dev data first. AI-generated SQL is confident and frequently subtly wrong (join fan-out, NULL handling, timezone logic).
  • Use Cowork for code that touches our files. When the script must read actual project data, write and run it in Claude Cowork against the task folder — not in an external coding tool.

Token economics

Routing query generation to a free or low-cost coding agent saves Claude capacity twice: the generation itself, and the iteration loop (most queries take 2–3 refinements). Reserve Claude for queries that need engagement context — "flag the pattern we discussed for this client" — which an external agent has no business seeing anyway.

This routing table is a starting point, not a substitute for the tier classification on Data Security. When a task's tier is unclear, treat it as the more restrictive tier and ask your project lead rather than guessing toward the cheaper tool.

ชุดเครื่องมือ AI ที่กว้างขึ้น

LLM สาธารณะฟรีและตัวแทนเขียนโค้ดเฉพาะทางช่วยให้เราสงวนขีดความสามารถของ Claude แบบชำระเงินไว้สำหรับงานที่ต้องการจริง ๆ ข้อแลกเปลี่ยนคือเรื่องการควบคุม — ยิ่งเครื่องมือราคาถูกเท่าใด เนื้อหาที่แตะต้องได้ก็ยิ่งต้องมีความอ่อนไหวน้อยลงเท่านั้น

LLM สาธารณะฟรี: DeepSeek & Kimi

DeepSeek (DeepSeek AI) และ Kimi (Moonshot AI) เป็นโมเดลฟรีที่มีความสามารถสูง — แข็งแกร่งด้านการเขียนโค้ด การจัดการเอกสารยาว และการร่างเนื้อหาทั่วไป มักทัดเทียมกับโมเดลแบบชำระเงินในงานทั่วไป เป็นเครื่องมือประหยัดโทเคนที่ยอดเยี่ยมสำหรับงาน Tier 1

ข้อจำกัดเด็ดขาด: Tier 1 เท่านั้น บริการเหล่านี้โฮสต์อยู่นอกการควบคุมของเรา ด้วยเงื่อนไขการจัดการข้อมูลที่เราไม่สามารถบังคับใช้ได้ — และเมื่อพิจารณาว่า ENVISAGE ดำเนินงานที่ใดและเราวิจัยเรื่องอะไร เนื้อหาของเรามีความอ่อนไหวเป็นพิเศษในบริบทนั้น ห้ามส่งข้อมูลภายใน ข้อมูลที่เกี่ยวข้องกับลูกค้า หรือชื่อจากงานวิจัยของเรา — ไม่มีข้อยกเว้นเด็ดขาด ให้ถือว่าทุกสิ่งที่ส่งไปจะถูกเก็บไว้และสามารถอ่านได้

การใช้งานที่เหมาะสม (Tier 1)

  • สรุปข่าวและรายงานสาธารณะที่เผยแพร่แล้วระหว่างการคัดกรองข้อมูล open-source
  • การร่างเนื้อหาทั่วไปในขั้นแรก: คำอธิบาย boilerplate วาระการประชุม เทมเพลตการติดต่อทั่วไป
  • อธิบายแนวคิดสาธารณะ วิธีการทางสถิติ หรือไลบรารี ก่อนที่จะใช้โทเคนของ Claude ไปกับการนำไปประยุกต์ใช้กับบริบทของเรา
  • ตัวอย่างโค้ดทั่วไปและความช่วยเหลือด้านการดีบัก (ไม่มีชื่อ schema ไม่มีโค้ดภายใน)
  • ระดมความคิดเรื่องโครงสร้างและ checklist ที่ไม่มีรายละเอียดเฉพาะของ ENVISAGE
การจัดเส้นทางที่ถูกต้อง

นักวิเคราะห์ที่กำลังคัดกรองบทความสาธารณะ 30 บทความขอให้ Kimi สรุปแต่ละบทความเป็นสองประโยค เลือก 6 บทความที่เกี่ยวข้อง จากนั้นนำเฉพาะบทความเหล่านั้นเข้า Claude เพื่อวิเคราะห์บริบทกับงานที่เกี่ยวข้อง เครื่องมือฟรีทำหน้าที่อ่านจำนวนมาก ส่วน Claude ทำหน้าที่ตัดสินใจ

ห้ามเด็ดขาด

การวางร่างการประเมินลูกค้าลงใน DeepSeek "แค่เพื่อขัดเกลาถ้อยคำ" ถ้อยคำนั้นผูกติดอยู่กับเนื้อหา Tier 3 — ร่างนั้นจึงตกอยู่บนโครงสร้างพื้นฐานที่เราไม่ได้ควบคุม

ตัวแทนเขียนโค้ด: จากภาษาธรรมดาสู่ SQL และอื่น ๆ

ตัวแทนแปลงภาษาธรรมดาเป็น SQL (เครื่องมือ NL2SQL แบบ Pi, SQL Agent CLI และเครื่องมือที่คล้ายกัน) และผู้ช่วยเขียนโค้ดแปลงคำขอภาษาอังกฤษทั่วไปให้กลายเป็นคำสั่ง query และสคริปต์ที่ใช้งานได้จริง สำหรับทีมที่ใช้ stack เป็น Python, PostgreSQL, Elasticsearch และ Snowflake เครื่องมือเหล่านี้ช่วยให้นักวิเคราะห์ที่เขียน SQL ไม่เป็นสามารถหาคำตอบได้โดยไม่ต้องรอคิวนักพัฒนา

รูปแบบที่ปลอดภัย: schema เข้า query ออก

ตัวแทนเขียนโค้ดไม่จำเป็นต้องใช้ข้อมูลของคุณเพื่อเขียน query — มันต้องการเพียง รูปร่าง ของข้อมูลของคุณ อธิบายตารางและคอลัมน์ แล้วรัน query ที่สร้างขึ้นด้วยตัวคุณเองภายในสภาพแวดล้อมของเรา

Our PostgreSQL database has:
  entities(entity_id, name_hash, type, country, first_seen)
  relationships(rel_id, entity_a, entity_b, rel_type, confidence)
  mentions(mention_id, entity_id, source_id, published_date)

Write a query returning entities mentioned in 3+ sources
in the last 90 days that have at least one 'ownership'
relationship, ordered by mention count.

สังเกตสิ่งที่อยู่ใน prompt: ชื่อคอลัมน์และเจตนา — ไม่มีแถวข้อมูล ไม่มีชื่อจริง ไม่มีข้อมูลรับรอง ตัวแทนจะส่งคืน SQL ให้คุณตรวจสอบ ตรวจสอบไขว้ หากเกี่ยวข้องกับ pipeline แล้วจึงรันภายใน Sapphire

กฎสำหรับตัวแทนเขียนโค้ด

  • โครงสร้าง schema เท่านั้น ชื่อตารางและคอลัมน์เป็น Tier 2 — ใช้กับเครื่องมือที่ได้รับอนุมัติได้ แถวข้อมูลจะอยู่ใน tier ใดก็ตามที่ข้อมูลนั้นเป็น และมักเป็น Tier 3
  • ห้ามวางข้อมูลรับรอง หรือ connection string และห้ามให้ตัวแทนภายนอกเชื่อมต่อโดยตรงกับฐานข้อมูล production คำสั่ง query ที่สร้างขึ้นต้องรันผ่านการควบคุมการเข้าถึงของเราเอง
  • ทดสอบกับข้อมูล dev ก่อน SQL ที่ AI สร้างขึ้นมีความมั่นใจสูงแต่มักผิดพลาดแบบแนบเนียน (join fan-out, การจัดการ NULL, ตรรกะ timezone)
  • ใช้ Cowork สำหรับโค้ดที่แตะต้องไฟล์ของเรา เมื่อสคริปต์ต้องอ่านข้อมูลโครงการจริง ให้เขียนและรันใน Claude Cowork กับโฟลเดอร์งาน — ไม่ใช่ในเครื่องมือเขียนโค้ดภายนอก

เศรษฐศาสตร์โทเคน

การจัดเส้นทางการสร้าง query ไปยังตัวแทนเขียนโค้ดฟรีหรือราคาถูกช่วยประหยัดขีดความสามารถของ Claude ได้สองต่อ: การสร้างเอง และรอบการทำซ้ำ (query ส่วนใหญ่ต้องปรับ 2–3 รอบ) สงวน Claude ไว้สำหรับ query ที่ต้องการบริบทของงาน — เช่น "ระบุรูปแบบที่เราคุยกันไว้สำหรับลูกค้ารายนี้" — ซึ่งตัวแทนภายนอกไม่ควรเห็นอยู่แล้ว

กฎการจัดการข้อมูลยังคงมีผลบังคับใช้

ตารางการจัดเส้นทางนี้เป็นเพียงจุดเริ่มต้น ไม่ใช่สิ่งทดแทนการจัดชั้นข้อมูลในหน้า ความปลอดภัยข้อมูล เมื่อ tier ของงานไม่ชัดเจน ให้ถือว่าเป็น tier ที่เข้มงวดกว่าและสอบถามหัวหน้าโครงการของคุณ แทนที่จะเดาไปทางเครื่องมือที่ถูกกว่า

အပိုင်း 5

ကျယ်ပြန့်သော AI ကိရိယာစုံ

အခမဲ့ အများသူငှာ LLM များနှင့် အထူးပြု ကုဒ်ရေးသား agent များက ကျွန်ုပ်တို့အား Claude ၏ ငွေပေးချေရသော စွမ်းရည်ကို လိုအပ်သောအလုပ်များအတွက် သီးသန့်ချန်ထားနိုင်စေသည်။ အပြန်အလှန်ဖလှယ်မှုမှာ ထိန်းချုပ်မှုပင်ဖြစ်သည် — ကိရိယာ ပိုစျေးသက်သာလေ ၎င်းက ထိတွေ့နိုင်သည့် အကြောင်းအရာ ပိုအောင်ဖျင်းလေဖြစ်သည်။

အခမဲ့ အများသူငှာ LLM များ: DeepSeek & Kimi

DeepSeek (DeepSeek AI) နှင့် Kimi (Moonshot AI) တို့သည် စွမ်းရည်ကောင်းသော အခမဲ့ model များဖြစ်သည် — ကုဒ်ရေးရေး၊ ရှည်လျားသောစာတမ်း ကိုင်တွယ်မှု နှင့် ယေဘုယျ မူကြမ်းရေးသားမှုတွင် အားကောင်းပြီး ယေဘုယျအလုပ်များတွင် ငွေပေးချေရသော model များနှင့် မကြာခဏ ယှဉ်ပြိုင်နိုင်သည်။ Tier 1 လုပ်ငန်းများအတွက် token သက်သာစေသော ထူးချွန်ကိရိယာများဖြစ်သည်။

လုံးဝကန့်သတ်ချက်: Tier 1 သာ။ ဤဝန်ဆောင်မှုများသည် ကျွန်ုပ်တို့၏ ထိန်းချုပ်မှုပြင်ပတွင် host လုပ်ထားပြီး ကျွန်ုပ်တို့ အတည်ဖြစ်စေနိုင်ခြင်းမရှိသော ဒေတာကိုင်တွယ်မှု စည်းကမ်းများနှင့်အတူဖြစ်သည် — ENVISAGE မည်သည့်နေရာတွင် လုပ်ဆောင်နေသည်နှင့် ကျွန်ုပ်တို့ မည်သည့်အရာကို သုတေသနလုပ်နေသည်ကို ထည့်သွင်းစဉ်းစားလျှင် ကျွန်ုပ်တို့၏ ကိစ္စရပ်သည် ထိုအခြေအနေတွင် အထူးအရေးကြီးသည်။ အတွင်းပိုင်းအရာ၊ ဖောက်သည်ဆက်စပ်အရာ၊ ကျွန်ုပ်တို့၏သုတေသနမှ နာမည်များ — မည်သည့်အခါမျှ မပို့ရ။ ပို့လိုက်သမျှ အရာအားလုံးကို သိမ်းဆည်းထားပြီး ဖတ်နိုင်သည်ဟု ယူဆပါ။

ကောင်းသော အသုံးပြုမှုများ (Tier 1)

  • open-source ဒေတာ ကောက်ယူစစ်ဆေးနေစဉ် ထုတ်ဝေပြီးသား သတင်းဆောင်းပါးများနှင့် အများသူငှာ အစီရင်ခံစာများကို အကျဉ်းချုပ်ခြင်း
  • ပထမအဆင့် ယေဘုယျ မူကြမ်းရေးသားခြင်း — boilerplate ဖော်ပြချက်များ၊ အစည်းအဝေးလုပ်ငန်းစဉ်များ၊ ယေဘုယျ ဆက်သွယ်ရေး template များ
  • Claude ၏ token ကို ကျွန်ုပ်တို့ဆက်စပ်နေရာတွင် အသုံးမချမီ အများသူငှာသဘောတရား၊ statistical method သို့မဟုတ် library ကို ရှင်းပြခြင်း
  • ယေဘုယျ code နမူနာနှင့် debugging အကူအညီ (schema အမည်မပါ၊ အတွင်းပိုင်း code မပါ)
  • ENVISAGE အသေးစိတ်မပါသော ဖွဲ့စည်းပုံများနှင့် checklist များကို idea ရှာခြင်း
မှန်ကန်သော routing

အများသူငှာ ဆောင်းပါး 30 ခုကို စစ်ဆေးနေသော analyst သည် Kimi ကို ဆောင်းပါးတစ်ခုချင်းစီအတွက် နှစ်စာကြောင်း အကျဉ်းချုပ်တောင်းပြီး၊ သက်ဆိုင်သော 6 ခုကို ရွေးချယ်ကာ ထိုအရာများကိုသာ Claude သို့ ယူဆောင်ကာ engagement နှင့်ဆက်စပ်၍ ခွဲခြမ်းစိတ်ဖြာသည်။ အခမဲ့ကိရိယာက အများအပြားဖတ်ရှုမှုကို လုပ်ဆောင်ပြီး Claude က ဆုံးဖြတ်ချက်ချမှုကို ကိုင်တွယ်သည်။

ဘယ်တော့မှ

ဖောက်သည် အကဲဖြတ်မူကြမ်းကို "စကားလုံးအနည်းငယ် ချောမွေ့စေရန်" ဆိုပြီး DeepSeek ထဲသို့ ကူးထည့်ခြင်း။ ထိုစကားလုံးများသည် Tier 3 အကြောင်းအရာနှင့် ချိတ်ဆက်နေသောကြောင့် ထိုမူကြမ်းသည် ကျွန်ုပ်တို့ ထိန်းချုပ်မှုမရှိသော infrastructure ပေါ်တွင် ကျန်ရှိသွားပါပြီ။

ကုဒ်ရေးသား Agent များ: သဘာဝဘာသာစကားမှ SQL နှင့် ထို့ထက်ပို၍

သဘာဝဘာသာစကားမှ SQL သို့ ပြောင်းလဲပေးသော agent များ (Pi-စတိုင် NL2SQL ကိရိယာများ၊ SQL Agent CLI နှင့် ဆင်တူသော ကိရိယာများ) နှင့် ကုဒ်ရေးသား ကူညီသူများသည် ရိုးရိုးအင်္ဂလိပ် တောင်းဆိုချက်များကို အလုပ်လုပ်နိုင်သော query နှင့် script များအဖြစ် ပြောင်းလဲပေးသည်။ Python, PostgreSQL, Elasticsearch နှင့် Snowflake ပေါ်တွင် stack run နေသော team တစ်ခုအတွက် SQL မရေးတတ်သော analyst များသည် developer ကို queue မစောင့်ဘဲ အဖြေရနိုင်စေသည်။

လုံခြုံသော pattern: schema ဝင်၊ query ထွက်

ကုဒ်ရေးသား agent သည် query ရေးရန် သင်၏ဒေတာကို လိုအပ်ခြင်း မရှိပါ — ၎င်းသည် သင်၏ဒေတာ၏ ပုံသဏ္ဌာန် ကိုသာ လိုအပ်သည်။ table နှင့် column များကို ဖော်ပြပြီး ဖန်တီးထားသော query ကို ကျွန်ုပ်တို့၏ ပတ်ဝန်းကျင်တွင် ကိုယ်တိုင် run ပါ။

Our PostgreSQL database has:
  entities(entity_id, name_hash, type, country, first_seen)
  relationships(rel_id, entity_a, entity_b, rel_type, confidence)
  mentions(mention_id, entity_id, source_id, published_date)

Write a query returning entities mentioned in 3+ sources
in the last 90 days that have at least one 'ownership'
relationship, ordered by mention count.

prompt ထဲတွင် ပါဝင်သည်များကို သတိပြုပါ — column အမည်များနှင့် ရည်ရွယ်ချက် — data row မပါ၊ တကယ့်နာမည် မပါ၊ credential မပါ။ Agent သည် SQL ကို ပြန်ပေးသည်။ သင် ၎င်းကို ပြန်လည်စစ်ဆေးပြီး pipeline နှင့်ဆက်စပ်ပါက ပြန်လည်စစ်ဆေး ကာ Sapphire အတွင်း execute လုပ်ပါ။

ကုဒ်ရေးသား agent များအတွက် စည်းမျဉ်းများ

  • Schema ဖွဲ့စည်းပုံသာ Table နှင့် column အမည်များသည် Tier 2 ဖြစ်သည် — ခွင့်ပြုထားသော ကိရိယာများနှင့် ကောင်းမွန်သည်။ Data row များမူ ဒေတာမည်သည့် tier ဖြစ်သည် ဆိုသည့်အလျောက်ဖြစ်ပြီး များသောအားဖြင့် Tier 3 ဖြစ်သည်။
  • Credential များ ဘယ်တော့မှ မကူးထည့်ပါနှင့် Connection string လည်း မထည့်ပါနှင့်၊ ပြင်ပ agent ကို production database များနှင့် တိုက်ရိုက် ချိတ်ဆက်ခွင့် မပြုပါနှင့်။ ဖန်တီးထားသော query များသည် ကျွန်ုပ်တို့၏ ကိုယ်ပိုင် access control များမှတစ်ဆင့် run ရမည်။
  • Dev data ဖြင့် ဦးစွာစမ်းသပ်ပါ AI ဖန်တီးထားသော SQL သည် ယုံကြည်ချက်ရှိသော်လည်း သိမ်ငယ်စွာ မှားယွင်းလေ့ရှိသည် (join fan-out, NULL ကိုင်တွယ်မှု၊ timezone logic)
  • ကျွန်ုပ်တို့ ဖိုင်များကို ထိတွေ့သော code အတွက် Cowork ကို သုံးပါ Script က တကယ့်ပရောဂျက် ဒေတာကို ဖတ်ရမည့်အခါ၊ task folder ကို ဆန့်ကျင်၍ Claude Cowork တွင် ရေးပြီး run ပါ — ပြင်ပ ကုဒ်ရေးသား ကိရိယာတွင် မဟုတ်ပါ။

Token စီးပွားရေး

Query ဖန်တီးမှုကို အခမဲ့ သို့မဟုတ် ဈေးသက်သာသော ကုဒ်ရေးသား agent သို့ routing လုပ်ခြင်းသည် Claude ၏ စွမ်းရည်ကို နှစ်ကြိမ် သက်သာစေသည် — ဖန်တီးမှုကိုယ်တိုင်နှင့် ထပ်ခါထပ်ခါ ပြန်လည်ပြင်ဆင်မှု (query အများစုသည် 2–3 ကြိမ် ပြန်လည်ပြင်ဆင်ရသည်)။ engagement ဆက်စပ်မှု လိုအပ်သော query များအတွက် Claude ကို ချန်ထားပါ — ဥပမာ "ဤဖောက်သည်အတွက် ကျွန်ုပ်တို့ ဆွေးနွေးခဲ့သော pattern ကို ဖော်ပြပါ" — ၎င်းကို ပြင်ပ agent က မမြင်သင့်သည့် ကိစ္စဖြစ်သည်။

Routing Cheat Sheet

အလုပ်ပထမရွေးချယ်မှုအကြောင်းရင်း
အများသူငှာ ဆောင်းပါးများကို အများအပြား အကျဉ်းချုပ်ခြင်းDeepSeek / KimiTier 1၊ ပမာဏများ — အခမဲ့ကိရိယာများက token ဝန်ကို စုပ်ယူသည်
Engagement ဆက်စပ်မှုကို အသုံးပြု၍ ခွဲခြမ်းစိတ်ဖြာခြင်းClaude (Project)ကျွန်ုပ်တို့၏ နည်းလမ်းနှင့် လျှို့ဝှက်ထိန်းသိမ်းမှု လိုအပ်သည်
Deliverable ဖိုင်များ တည်ဆောက်ခြင်း သို့မဟုတ် တည်းဖြတ်ခြင်းClaude Coworkထိန်းချုပ်ထားသော task folder တွင် တိုက်ရိုက်ဖိုင်လုပ်ငန်း
Pipeline အတွက် သဘာဝဘာသာစကား → SQLNL2SQL ကုဒ်ရေးသား agentSchema-only prompt၊ ဈေးသက်သာစွာ ထပ်ခါထပ်ခါလုပ်နိုင်
အချက်အလက်၊ ကိန်းဂဏန်း သို့မဟုတ် ဘာသာပြန်ချက် စစ်ဆေးခြင်းGemini / ChatGPT (အခမဲ့)context မပါသော တိုတောင်းသော စစ်ဆေးမှု prompt
ယေဘုယျ code ရှင်းပြခြင်း/debugDeepSeek / Kimiကုဒ်ရေးစွမ်းရည်ကောင်း၊ ကုန်ကျစရိတ်မရှိ၊ Tier 1 အကြောင်းအရာ
လူပုဂ္ဂိုလ်၊ ဖောက်သည် သို့မဟုတ် ရင်းမြစ်အမည်ဖော်ပြသော မည်သည့်အရာမဆိုClaude သာ — အမည်ဖျောက်ထားသောTier 3 သည် ခွင့်ပြုထားသော ကိရိယာများမှ ဘယ်တော့မှ မထွက်ရ၊ ideal အားဖြင့် anonymization မလုပ်ဘဲ ထိုကိရိယာများကိုပင် မသုံးသင့်

ဒေတာကိုင်တွယ်မှု စည်းမျဉ်းများ ဆက်လက်သက်ရောက်နေသည်

ဤ routing table သည် အစပြုနေရာတစ်ခုသာဖြစ်ပြီး ဒေတာလုံခြုံရေး စာမျက်နှာရှိ tier သတ်မှတ်ချက်၏ အစားထိုးမဟုတ်ပါ။ အလုပ်တစ်ခု၏ tier မရှင်းလင်းသောအခါ ပိုတင်းကျပ်သော tier အဖြစ် သတ်မှတ်ပြီး ဈေးသက်သာသော ကိရိယာဘက်သို့ မှန်းဆမည့်အစား သင့် project lead ကို မေးမြန်းပါ။
