Citation Source ID Bug Fix

A follow-up fix for citations loaded from saved chat sessions, where deserialized dictionaries — not citation objects — caused a crash when the code tried to read a source ID.

The Problem

When a chat session is reloaded from the database or a checkpoint, citations come back as plain dictionaries rather than citation objects. Code that assumed source.source_id was always an attribute failed on those dictionaries with a 'dict' object has no attribute 'source_id' error.

The Fix

  • Citation validation now handles both citation objects and dictionaries when extracting source IDs.
  • Error state now clears at session initialization, so a past error can't persist across sessions.
  • The citation processing logic was extracted into a dedicated helper, cutting the enclosing function's complexity from 53 statements / 13 branches down to roughly 25 statements / 8 branches — resolving the linter's complexity warnings in the process.

Reliability

New tests cover citation objects, dictionary citations (the fix case), mixed collections of both, pre-processed citations, and error handling. All existing tests continue to pass, and linting is clean.

Data Handling Still Applies

A backend deserialization fix — no change to what citations may reference or how tiered data flows through the system. See Data Security for the underlying rules.
