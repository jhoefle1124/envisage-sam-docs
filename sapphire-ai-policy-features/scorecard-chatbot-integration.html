Scorecard Rankings in Portfolio Analyst

The engineering behind the scorecard_rankings query type covered on the Scorecard Rankings feature page — batch scorecard retrieval jumped from 3 to 50 companies per query, and factor-based rankings became a single tool call instead of one call per company.

What Changed

  • New query type: scorecard_rankings ranks portfolio or watchlist companies by internal, external, or ESG scorecard factors in one request.
  • Batch processing: up to 50 companies per query, replacing N separate tool calls with a single one — with smart input handling for single queries, multiple queries, or direct company IDs.
  • PostgreSQL fix: resolved DISTINCT ON clause issues by moving to proper subqueries, eliminating a class of N+1 query problems.

Architecture

The previously monolithic portfolio analyst was broken into organized modules following a repository pattern, with a query builder abstraction for maintainable, extensible query construction. See Scorecard Rankings for the user-facing view of this feature.

Reliability

100+ test cases across 14 test modules, including integration tests for complete workflows, edge cases for batch operations, and performance validation for the N+1 query fixes. Full backward compatibility maintained.

Data Handling Still Applies

Batch scorecard retrieval doesn't change what tier the data carries — portfolio and scorecard data stays Tier 3 per Data Security regardless of how many companies are in a single query.

การจัดอันดับ Scorecard ใน Portfolio Analyst

เบื้องหลังด้านวิศวกรรมของ query type ประเภท scorecard_rankings ที่กล่าวถึงในหน้าฟีเจอร์ Scorecard Rankings — การดึงข้อมูล scorecard แบบ batch เพิ่มขึ้นจาก 3 เป็น 50 บริษัทต่อ query และการจัดอันดับตาม factor กลายเป็นการเรียก tool เพียงครั้งเดียว แทนที่จะเรียกทีละบริษัท

การเปลี่ยนแปลงที่เกิดขึ้น

  • Query type ใหม่: scorecard_rankings จัดอันดับบริษัทในพอร์ตหรือ watchlist ตาม scorecard factor ทั้งภายใน ภายนอก หรือ ESG ได้ในคำขอเดียว
  • การประมวลผลแบบ batch: สูงสุด 50 บริษัทต่อ query แทนที่การเรียก tool แยกกัน N ครั้งด้วยการเรียกเพียงครั้งเดียว — พร้อมการจัดการ input อย่างชาญฉลาดสำหรับ query เดี่ยว, query หลายรายการ, หรือ company ID โดยตรง
  • การแก้ไข PostgreSQL: แก้ปัญหาของ clause DISTINCT ON โดยเปลี่ยนไปใช้ subquery ที่ถูกต้อง ซึ่งช่วยขจัดปัญหา N+1 query ทั้งกลุ่ม

สถาปัตยกรรม

Portfolio analyst ที่แต่เดิมมีโครงสร้างแบบ monolithic ถูกแยกออกเป็นโมดูลที่จัดระเบียบตามแนวทาง repository pattern พร้อมด้วย query builder abstraction เพื่อให้การสร้าง query ดูแลรักษาง่ายและขยายต่อได้ ดูหน้า Scorecard Rankings สำหรับมุมมองของฟีเจอร์นี้ในฝั่งผู้ใช้งาน

ความน่าเชื่อถือ

มี test case มากกว่า 100 รายการ ครอบคลุม 14 test module รวมถึง integration test สำหรับ workflow แบบครบวงจร, edge case สำหรับการทำงานแบบ batch และการตรวจสอบประสิทธิภาพสำหรับการแก้ไขปัญหา N+1 query ยังคงรักษาความเข้ากันได้แบบย้อนหลัง (backward compatibility) ไว้อย่างสมบูรณ์

หลักการจัดการข้อมูลยังคงใช้บังคับ

การดึงข้อมูล scorecard แบบ batch ไม่ได้เปลี่ยนแปลงระดับชั้น (tier) ของข้อมูล — ข้อมูลพอร์ตและ scorecard ยังคงเป็น Tier 3 ตามนโยบาย Data Security ไม่ว่า query เดียวจะมีจำนวนบริษัทมากเท่าใดก็ตาม

Portfolio Analyst တွင် Scorecard အဆင့်သတ်မှတ်ချက်များ

Scorecard Rankings ဖီချာစာမျက်နှာတွင် ဖော်ပြထားသော scorecard_rankings query အမျိုးအစား၏ နောက်ကွယ်ရှိ engineering — batch scorecard ထုတ်ယူမှုသည် query တစ်ခုလျှင် ကုမ္ပဏီ 3 ခုမှ 50 ခုအထိ တိုးလာပြီး၊ factor အခြေခံ အဆင့်သတ်မှတ်ချက်များသည် ကုမ္ပဏီတစ်ခုစီအတွက် tool ခေါ်ဆိုမှုတစ်ခုစီအစား tool ခေါ်ဆိုမှု တစ်ကြိမ်တည်းသာ ဖြစ်လာသည်။

ပြောင်းလဲမှုများ

  • Query အမျိုးအစားအသစ်: scorecard_rankings သည် ပေါ်တ်ဖိုလီယို သို့မဟုတ် watchlist ကုမ္ပဏီများကို အတွင်းပိုင်း၊ ပြင်ပ သို့မဟုတ် ESG scorecard factor များအလိုက် တောင်းဆိုမှုတစ်ခုတည်းဖြင့် အဆင့်သတ်မှတ်ပေးသည်။
  • Batch လုပ်ဆောင်ချက်: query တစ်ခုလျှင် ကုမ္ပဏီ 50 ခုအထိ — tool ခေါ်ဆိုမှု N ကြိမ် သီးခြားစီအစား တစ်ကြိမ်တည်းဖြင့် အစားထိုးသည်၊ query တစ်ခုတည်း၊ query များစွာ သို့မဟုတ် company ID တိုက်ရိုက်ဖြင့် ရိုက်ထည့်ခြင်းတို့အတွက် ဉာဏ်ရည်ရှိသော input ကိုင်တွယ်မှုနှင့်တကွ။
  • PostgreSQL ပြင်ဆင်ချက်: DISTINCT ON clause ပြဿနာများကို subquery များကို မှန်ကန်စွာ အသုံးပြုခြင်းဖြင့် ဖြေရှင်းပြီး N+1 query ပြဿနာအမျိုးအစားတစ်ခုလုံးကို ဖယ်ရှားလိုက်သည်။

ဗိသုကာပုံစံ

ယခင်က monolithic ဖြစ်နေသော portfolio analyst ကို repository pattern အတိုင်း စနစ်တကျ module များအဖြစ် ခွဲထုတ်လိုက်ပြီး၊ ထိန်းသိမ်းရလွယ်ကူပြီး တိုးချဲ့နိုင်သော query တည်ဆောက်မှုအတွက် query builder abstraction ပါဝင်သည်။ ဤဖီချာ၏ အသုံးပြုသူဘက်မှ ရှုထောင့်အတွက် Scorecard Rankings ကို ကြည့်ပါ။

ယုံကြည်စိတ်ချရမှု

Test module 14 ခုတွင် test case 100 ကျော် ပါဝင်ပြီး၊ ပြီးပြည့်စုံသော workflow များအတွက် integration test များ၊ batch လုပ်ဆောင်ချက်များအတွက် edge case များ နှင့် N+1 query ပြင်ဆင်ချက်များအတွက် စွမ်းဆောင်ရည် အတည်ပြုမှုများ ပါဝင်သည်။ Backward compatibility ကို အပြည့်အဝ ထိန်းသိမ်းထားသည်။

ဒေတာကိုင်တွယ်မှု စည်းမျဉ်းများ ဆက်လက်သက်ရောက်

Batch scorecard ထုတ်ယူမှုသည် ဒေတာ၏ tier ကို ပြောင်းလဲစေခြင်း မရှိပါ — Data Security အရ ပေါ်တ်ဖိုလီယိုနှင့် scorecard ဒေတာသည် query တစ်ခုတွင် ကုမ္ပဏီအရေအတွက် မည်မျှပါဝင်သည်ဖြစ်စေ Tier 3 အဖြစ် ဆက်လက်ရှိနေမည်ဖြစ်သည်။
