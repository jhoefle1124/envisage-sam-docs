New in SAM · Feature Guide

Enhanced Export Functionality

Exporting screener results is now more robust: clearer error messages, one shared export service behind both the AI screener and the manual screener, and export limits that reflect the actual configured maximum instead of a hardcoded number.

What's New

  • Better error handling: export failures now surface user-friendly, actionable messages instead of generic errors.
  • Shared service layer: AI screener and manual screener exports now run through one consolidated service, instead of separate, duplicated logic.
  • Dynamic export limits: the export limit message reflects the actual configured maximum, not a hardcoded number.
  • Screener name validation: names over 60 characters now get a clear error message instead of failing silently.

Why It Matters

Before this change, the AI screener and manual screener each had their own export logic, which meant fixes and improvements had to be made twice — and sometimes only made it into one path. Consolidating both into a shared service means a more consistent experience regardless of which screener you're exporting from, and one place to maintain going forward.

Error Handling Improvements

ScenarioWhat Happens Now
Screener name longer than 60 charactersClear, specific error message instead of a failed or silent export
Export limit reachedMessage states the actual configured limit, read from the SCREENER_EXPORT_LIMIT environment variable
Other export failuresUser-friendly, actionable error messages instead of generic failures

Reliability

Thoroughly tested, no breaking changes. The update ships with 345 lines of new tests for export helper functions and 539 lines of tests for AI screener forms integration, plus fixes for previously flaky tests. All existing export functionality is preserved.

Data Handling Still Applies

Exporting results built from public screening criteria is generally Tier 1. The tier shifts once an export carries client-specific framing.

Usually fine

Public screen exports

Exporting a generic screen built on public criteria, with no client name or engagement context attached, stays close to Tier 1.

Check first

Named or client-tied exports

If a saved screener's name, notes, or export destination reference a specific client or engagement, treat that export as Tier 2/3 and route it through approved workflows, per Data Security.

การปรับปรุงฟังก์ชันการส่งออกข้อมูล

การส่งออกผลลัพธ์จากตัวคัดกรอง (screener) มีความเสถียรมากขึ้น: ข้อความแจ้งข้อผิดพลาดที่ชัดเจนขึ้น มี export service ที่ใช้ร่วมกันเบื้องหลังทั้ง AI screener และ manual screener และขีดจำกัดการส่งออกที่สะท้อนค่าสูงสุดที่ตั้งค่าไว้จริง แทนที่จะเป็นตัวเลขที่ hardcode ไว้ตายตัว

มีอะไรใหม่

  • การจัดการข้อผิดพลาดที่ดีขึ้น: เมื่อการส่งออกล้มเหลว ระบบจะแสดงข้อความที่เป็นมิตรกับผู้ใช้และสามารถนำไปปฏิบัติได้ แทนที่จะเป็นข้อความ error ทั่วไป
  • Service layer ที่ใช้ร่วมกัน: การส่งออกของทั้ง AI screener และ manual screener ทำงานผ่าน service ที่รวมเป็นหนึ่งเดียวแล้ว แทนที่จะใช้ logic ที่แยกกันและซ้ำซ้อน
  • ขีดจำกัดการส่งออกแบบไดนามิก: ข้อความขีดจำกัดการส่งออกสะท้อนค่าสูงสุดที่ตั้งค่าไว้จริง ไม่ใช่ตัวเลขที่ hardcode ไว้
  • การตรวจสอบชื่อ screener: ชื่อที่ยาวเกิน 60 ตัวอักษร จะได้รับข้อความแจ้งข้อผิดพลาดที่ชัดเจน แทนที่จะล้มเหลวแบบเงียบ ๆ

เหตุใดจึงสำคัญ

ก่อนการเปลี่ยนแปลงนี้ AI screener และ manual screener ต่างก็มี export logic ของตัวเอง ซึ่งหมายความว่าการแก้ไขและปรับปรุงต้องทำซ้ำสองครั้ง — และบางครั้งก็ถูกนำไปใช้แค่ใน path เดียวเท่านั้น การรวมทั้งสองเข้าเป็น service เดียวทำให้ประสบการณ์การใช้งานสอดคล้องกันมากขึ้นไม่ว่าจะส่งออกจาก screener ใด และมีจุดเดียวสำหรับดูแลรักษาต่อไปในอนาคต

การปรับปรุงการจัดการข้อผิดพลาด

สถานการณ์สิ่งที่เกิดขึ้นในตอนนี้
ชื่อ screener ยาวเกิน 60 ตัวอักษรข้อความแจ้งข้อผิดพลาดที่ชัดเจนและเจาะจง แทนที่จะเป็นการส่งออกที่ล้มเหลวหรือเงียบหาย
ถึงขีดจำกัดการส่งออกแล้วข้อความระบุขีดจำกัดที่ตั้งค่าไว้จริง โดยอ่านค่าจากตัวแปรสภาพแวดล้อม SCREENER_EXPORT_LIMIT
ความล้มเหลวในการส่งออกอื่น ๆข้อความแจ้งข้อผิดพลาดที่เป็นมิตรกับผู้ใช้และสามารถนำไปปฏิบัติได้ แทนที่จะเป็นความล้มเหลวทั่วไป

ความน่าเชื่อถือ

ผ่านการทดสอบอย่างละเอียดถี่ถ้วน ไม่มีการเปลี่ยนแปลงที่ทำให้ระบบเดิมพัง การอัปเดตนี้มาพร้อมกับ test ใหม่ 345 บรรทัดสำหรับฟังก์ชัน export helper และ test 539 บรรทัดสำหรับการทดสอบ integration ของฟอร์ม AI screener รวมถึงการแก้ไข test ที่เคย flaky มาก่อน ฟังก์ชันการส่งออกเดิมทั้งหมดยังคงทำงานตามปกติ

หลักการจัดการข้อมูลยังคงมีผลบังคับใช้

การส่งออกผลลัพธ์ที่สร้างจากเกณฑ์การคัดกรองสาธารณะโดยทั่วไปจัดเป็น Tier 1 ระดับชั้นจะเปลี่ยนไปเมื่อการส่งออกนั้นมีบริบทที่เจาะจงถึงลูกค้า

การส่งออก screen สาธารณะ

การส่งออก screen ทั่วไปที่สร้างจากเกณฑ์สาธารณะ โดยไม่มีชื่อลูกค้าหรือบริบทของงานที่เกี่ยวข้องแนบมาด้วย ยังคงใกล้เคียงกับ Tier 1

ตรวจสอบก่อน

การส่งออกที่ระบุชื่อหรือผูกกับลูกค้า

หากชื่อ screener ที่บันทึกไว้ หมายเหตุ หรือปลายทางการส่งออก อ้างอิงถึงลูกค้าหรืองานที่เจาะจง ให้ถือว่าการส่งออกนั้นเป็น Tier 2/3 และดำเนินการผ่าน workflow ที่ได้รับอนุมัติ ตามที่ระบุใน Data Security

Export လုပ်ဆောင်ချက် တိုးတက်မှု

Screener ရလဒ်များ export လုပ်ခြင်းသည် ယခုအခါ ပိုမိုခိုင်မာလာသည် — ပိုရှင်းလင်းသော error message များ၊ AI screener နှင့် manual screener နှစ်ခုစလုံးအနောက်တွင် တူညီသော export service တစ်ခုတည်း၊ နှင့် hardcode ထားသည့် ကိန်းဂဏန်းအစား တကယ် config လုပ်ထားသည့် အများဆုံးကန့်သတ်ချက်ကို ထင်ဟပ်သော export limit များ ပါဝင်လာသည်

အသစ်ပါဝင်သည့်အချက်များ

  • Error handling ပိုမိုကောင်းမွန်လာခြင်း: export မအောင်မြင်သောအခါ ယခုအခါ generic error များအစား သုံးစွဲသူအတွက် နားလည်လွယ်ပြီး လုပ်ဆောင်နိုင်သော message များ ပြသသည်
  • Service layer တစ်ခုတည်း အသုံးပြုခြင်း: AI screener နှင့် manual screener export များသည် ယခုအခါ သီးခြားနှင့် ထပ်နေသော logic များအစား တစ်ခုတည်းသော ပေါင်းစည်းထားသည့် service မှတစ်ဆင့် လုပ်ဆောင်သည်
  • Export limit ကို dynamically ပြသခြင်း: export limit message သည် hardcode ထားသည့် ကိန်းဂဏန်းအစား တကယ် config လုပ်ထားသည့် အများဆုံးကန့်သတ်ချက်ကို ထင်ဟပ်သည်
  • Screener နာမည် validation: ၆၀ လုံးထက်ကျော်သော နာမည်များသည် ယခုအခါ ဆိတ်ဆိတ်ချောက်ချားမှုအစား ရှင်းလင်းသော error message ရရှိသည်

အရေးကြီးရသည့်အကြောင်းရင်း

ဤပြောင်းလဲမှုမတိုင်မီ AI screener နှင့် manual screener တစ်ခုစီတွင် သီးခြား export logic ရှိခဲ့သည် — ဆိုလိုသည်မှာ ပြင်ဆင်မှုနှင့် တိုးတက်မှုများကို နှစ်ကြိမ် ပြုလုပ်ရပြီး တစ်ခါတစ်ရံ path တစ်ခုတည်းတွင်သာ ရောက်ရှိသည့်အခါလည်း ရှိသည်။ နှစ်ခုစလုံးကို service တစ်ခုတည်းအဖြစ် ပေါင်းစည်းလိုက်ခြင်းဖြင့် မည်သည့် screener မှ export လုပ်သည်ဖြစ်စေ ပိုမိုတသမတ်တည်းရှိသော အတွေ့အကြုံရရှိပြီး နောင်တွင် ထိန်းသိမ်းရန် နေရာတစ်ခုတည်းရှိလာသည်

Error Handling တိုးတက်မှုများ

အခြေအနေယခုဖြစ်ပေါ်သည့်အရာ
Screener နာမည် ၆၀ လုံးထက်ကျော်ခြင်းExport မအောင်မြင်ခြင်း သို့မဟုတ် ဆိတ်ဆိတ်ဖြစ်ခြင်းအစား ရှင်းလင်းပြီး တိကျသော error message
Export limit သို့ ရောက်ရှိခြင်းSCREENER_EXPORT_LIMIT environment variable မှ ဖတ်ယူထားသော တကယ့် config limit ကို message က ဖော်ပြသည်
အခြား export မအောင်မြင်မှုများGeneric failure များအစား သုံးစွဲသူအတွက် နားလည်လွယ်ပြီး လုပ်ဆောင်နိုင်သော error message များ

ယုံကြည်စိတ်ချရမှု

အသေးစိတ် စစ်ဆေးပြီး ရှိပြီးသား functionality မပျက်စီးပါ။ ဤ update သည် export helper function များအတွက် test line ၃၄၅ လိုင်းနှင့် AI screener form များ integration testing အတွက် line ၅၃၉ လိုင်း အသစ်များနှင့်အတူ ရောက်ရှိလာပြီး previously flaky ဖြစ်ခဲ့သော test များကို ပြင်ဆင်ပေးထားသည်။ ရှိပြီးသား export functionality အားလုံး ဆက်လက်ရပ်တည်နေသည်
