AI Core Runtime Extraction

A structural refactor: shared AI runtime infrastructure moved out of the chatbots app into a new apps.ai_core, and tool providers relocated to the domain apps that actually own their data — no functional behavior changes.

Why

The chatbots app had grown into a general-purpose AI platform module that other apps (AI screener, portfolios, scorecards) had to reach into for tool contracts, the LLM factory, and stock-screener query types. That coupling made chat-only changes risky and blocked reuse of the tool runtime outside the chat surface.

What Moved Where

New homeTool providers
apps.ai_core.tool_providersGeneric external integrations: Tavily search, Jina reader, Brave News search, broker notes MCP, Firecrawl
apps.stock_data.tool_providerscompany_financials, daily_price_metrics, price_equity
apps.stock_screener.tool_providersstock_screener tool and its complexity limiter
apps.scorecards.tool_providersscorecard
apps.portfolios.tool_providersportfolio_analyst
apps.chatbots.tool_providersStays: chat-only tools (document tools, chart_generator) tied to chat session state

A new apps.ai_core app now hosts the tool registry, context, responses, base classes, categories, LLM factory, exceptions, URL helpers, and financial formatter. Stock-screener query contracts (filter schemas, field registry, query builder) moved to apps.stock_screener.query.

Import-Boundary Enforcement

New AST-based regression tests guard the new architecture: runtime modules in ai_core and stock-screener query contracts may not import from apps.chatbots, each domain's tool provider package may not import from apps.chatbots, and apps.chatbots.tool_providers is guarded against new modules for already-migrated providers reappearing there.

Reliability

41 commits across four migration waves, with temporary compatibility shims removed once every runtime caller and test was repointed. The full test suite runs clean; the only behavioral change is a targeted performance fix in the stock screener's allowed-fields lookup, covered by its own dedicated test.

Data Handling Still Applies

A structural code refactor with no functional changes — every tool still respects the same tiering rules it did before the move. See Data Security for the underlying policy this codebase enforces.

การแยกส่วน AI Core Runtime

การปรับโครงสร้างโค้ด: โครงสร้างพื้นฐาน AI runtime ที่ใช้ร่วมกันถูกย้ายออกจากแอป chatbots ไปยัง apps.ai_core ที่สร้างขึ้นใหม่ และ tool providers ถูกย้ายไปยังแอปโดเมนที่เป็นเจ้าของข้อมูลนั้นจริง ๆ — ไม่มีการเปลี่ยนแปลงพฤติกรรมการทำงานใด ๆ

เหตุผล

แอป chatbots เติบโตขึ้นจนกลายเป็นโมดูลแพลตฟอร์ม AI อเนกประสงค์ ซึ่งแอปอื่น ๆ (AI screener, portfolios, scorecards) ต้องเข้ามาเรียกใช้เพื่อดึง tool contracts, LLM factory และ stock-screener query types การผูกติดกันเช่นนี้ทำให้การเปลี่ยนแปลงเฉพาะส่วนแชทมีความเสี่ยง และขัดขวางการนำ tool runtime ไปใช้ซ้ำนอกเหนือจากส่วนแชท

regression tests แบบใหม่ที่ใช้ AST คอยปกป้องสถาปัตยกรรมใหม่นี้ โดยกำหนดว่า runtime modules ใน ai_core และ stock-screener query contracts จะ import จาก apps.chatbots ไม่ได้ แพ็กเกจ tool provider ของแต่ละโดเมนก็จะ import จาก apps.chatbots ไม่ได้เช่นกัน และ apps.chatbots.tool_providers ถูกป้องกันไม่ให้มีโมดูลใหม่สำหรับ providers ที่ย้ายออกไปแล้วกลับมาปรากฏอีก

ความน่าเชื่อถือ

41 commits ตลอดสี่ระลอกของการย้ายระบบ (migration waves) โดย compatibility shims ชั่วคราวถูกลบออกทั้งหมดเมื่อ runtime caller และ test ทุกตัวถูกชี้ไปยังตำแหน่งใหม่เรียบร้อยแล้ว ชุดทดสอบทั้งหมดผ่านโดยไม่มีปัญหา การเปลี่ยนแปลงพฤติกรรมเพียงอย่างเดียวคือการแก้ไข performance แบบเจาะจงใน allowed-fields lookup ของ stock screener ซึ่งมี test เฉพาะของตัวเองคอยตรวจสอบ

หลักการจัดการข้อมูลยังคงบังคับใช้เหมือนเดิม

การปรับโครงสร้างโค้ดนี้ไม่มีการเปลี่ยนแปลงด้านฟังก์ชันการทำงาน — เครื่องมือทุกตัวยังคงปฏิบัติตามกฎการจัดชั้นข้อมูล (tiering rules) แบบเดียวกับก่อนการย้ายระบบ ดูรายละเอียดที่ Data Security สำหรับนโยบายที่โค้ดชุดนี้บังคับใช้

AI Core Runtime ခွဲထုတ်ခြင်း

ဖွဲ့စည်းပုံဆိုင်ရာ ပြုပြင်ပြောင်းလဲမှု — chatbots app မှ share ထားသော AI runtime အခြေခံအဆောက်အအုံကို အသစ်ဖန်တီးထားသော apps.ai_core ထဲသို့ ရွှေ့ပြီး၊ tool providers များကို ၎င်းတို့၏ဒေတာကို တကယ်ပိုင်ဆိုင်သော domain app များထဲသို့ ပြောင်းရွှေ့ခဲ့သည် — functional အပြုအမူ ပြောင်းလဲမှု မရှိပါ။

အကြောင်းရင်း

chatbots app သည် အခြား app များ (AI screener, portfolios, scorecards) က tool contracts, LLM factory နှင့် stock-screener query types များအတွက် ဝင်ရောက်အသုံးပြုရန် လိုအပ်သော ယေဘုယျ AI platform module တစ်ခုအဖြစ် ကြီးထွားလာခဲ့သည်။ ထိုချိတ်ဆက်မှုကြောင့် chat-only ပြောင်းလဲမှုများသည် အန္တရာယ်ရှိလာပြီး chat surface အပြင်ဘက်တွင် tool runtime ကို ပြန်လည်အသုံးပြုခြင်းကို ပိတ်ဆို့ထားသည်။

Import Boundary ကို တင်းကျပ်စွာ ကျင့်သုံးခြင်း

AST အခြေခံ regression test အသစ်များသည် ဗိသုကာအသစ်ကို ကာကွယ်ပေးသည် — ai_core ရှိ runtime module များနှင့် stock-screener query contracts များသည် apps.chatbots မှ import လုပ်ခွင့်မရှိပါ၊ domain တစ်ခုစီ၏ tool provider package များသည်လည်း apps.chatbots မှ import လုပ်ခွင့်မရှိပါ၊ ထို့အပြင် apps.chatbots.tool_providers ကို ရွှေ့ပြောင်းပြီးသား provider များအတွက် module အသစ်များ ပြန်လည်ပေါ်လာခြင်း မရှိစေရန် ကာကွယ်ထားသည်။

ယုံကြည်စိတ်ချရမှု

41 commits ဖြင့် migration wave လေးခုကို ဖြတ်သန်းခဲ့ပြီး၊ runtime caller နှင့် test အားလုံးကို ပြန်လည်ညွှန်းပြီးသည်နှင့် ယာယီ compatibility shim များကို ဖယ်ရှားခဲ့သည်။ test suite အပြည့်အစုံသည် ချောမွေ့စွာ run ဖြစ်သည်။ အပြုအမူ ပြောင်းလဲမှု တစ်ခုတည်းမှာ stock screener ၏ allowed-fields lookup တွင် ရည်ရွယ်ချက်ရှိရှိ performance ပြင်ဆင်မှုတစ်ခုဖြစ်ပြီး ၎င်းအတွက် သီးသန့် test တစ်ခုက အကာအကွယ်ပေးထားသည်။

ဒေတာကိုင်တွယ်မှု စည်းမျဉ်းများ ဆက်လက်သက်ရောက်နေဆဲ

ဖွဲ့စည်းပုံဆိုင်ရာ code ပြုပြင်ပြောင်းလဲမှုတစ်ခုသာဖြစ်ပြီး functional ပြောင်းလဲမှု မရှိပါ — ကိရိယာတိုင်းသည် ရွှေ့ပြောင်းမီကရှိခဲ့သည့် tiering rules များကိုပင် ဆက်လက်လိုက်နာနေဆဲဖြစ်သည်။ ဤ codebase မှ ကျင့်သုံးနေသော အခြေခံ policy အတွက် Data Security ကို ကြည့်ပါ။
