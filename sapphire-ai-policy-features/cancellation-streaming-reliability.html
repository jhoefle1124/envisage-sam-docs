Cancellation & Streaming Reliability

A reliability pass addressing concurrent requests, stale responses, and cancellation race conditions — plus a new stop button, longer streaming timeouts, and a fix for date filtering in the Brave News search tool.

Concurrency & Race Conditions

  • Concurrent response prevention: a new message is rejected while a response is already streaming for the same session, enforced with atomic database locks.
  • Stale response detection: trackers older than 10 minutes are automatically marked failed so they don't block new requests.
  • Cancellation improvements: flags for early cancellation (before a tracker exists) and duplicate-cancel prevention, with tracker terminal status checked before attempting to cancel.

Streaming & Timeouts

SettingBeforeAfter
Celery soft limit5 min12 min
Celery hard limit6 min15 min
Redis buffer TTL5 min15 min

A new stop button lets you cancel a response mid-generation, and stale responses (no updates for 15 minutes) now recover automatically instead of waiting indefinitely.

Other Fixes

  • Brave News search now uses the API's freshness parameter correctly, with support for preset codes and "X ago" style historical searches.
  • Nested line breaks in list items were removed so copying a response into Word or Google Docs no longer produces double-spaced lists.

Reliability

Test coverage for the affected consumer helpers expanded from 40% to 73% (19 → 57 tests), with new tests for stale detection, concurrent request prevention, and cancellation edge cases.

Data Handling Still Applies

A reliability and timing fix — no change to what data any tool call may access. See Data Security.

ฟีเจอร์ใหม่ใน SAM · เอกสารอ้างอิงทางเทคนิค

การยกเลิกคำขอและความเสถียรของการสตรีมข้อมูล

การปรับปรุงด้านความน่าเชื่อถือที่แก้ไขปัญหาคำขอพร้อมกัน (concurrent requests) การตอบกลับที่ค้าง (stale responses) และ race condition ในการยกเลิก — พร้อมปุ่มหยุดใหม่ ระยะเวลา timeout ของการสตรีมที่ยาวขึ้น และการแก้ไขการกรองตามวันที่ในเครื่องมือค้นหา Brave News

Concurrency และ Race Condition

  • การป้องกันการตอบกลับพร้อมกัน: ข้อความใหม่จะถูกปฏิเสธในขณะที่มีการสตรีมคำตอบอยู่แล้วสำหรับ session เดียวกัน โดยบังคับใช้ด้วย atomic database lock
  • การตรวจจับการตอบกลับที่ค้าง: tracker ที่เก่ากว่า 10 นาทีจะถูกทำเครื่องหมายว่าล้มเหลวโดยอัตโนมัติ เพื่อไม่ให้ไปขวางคำขอใหม่
  • การปรับปรุงการยกเลิก: เพิ่ม flag สำหรับการยกเลิกก่อนกำหนด (ก่อนที่ tracker จะถูกสร้างขึ้น) และการป้องกันการยกเลิกซ้ำ โดยตรวจสอบสถานะสุดท้าย (terminal status) ของ tracker ก่อนพยายามยกเลิก

การสตรีมข้อมูลและ Timeout

การตั้งค่าก่อนหน้าหลังปรับปรุง
Celery soft limit5 นาที12 นาที
Celery hard limit6 นาที15 นาที
Redis buffer TTL5 นาที15 นาที

ปุ่มหยุดใหม่ช่วยให้คุณยกเลิกการตอบกลับระหว่างที่กำลังสร้างเนื้อหาได้ และการตอบกลับที่ค้าง (ไม่มีการอัปเดตเป็นเวลา 15 นาที) ตอนนี้จะกู้คืนโดยอัตโนมัติแทนที่จะรอไม่มีกำหนด

การแก้ไขอื่น ๆ

  • Brave News search ตอนนี้ใช้พารามิเตอร์ freshness ของ API ได้อย่างถูกต้อง พร้อมรองรับรหัส preset และการค้นหาย้อนหลังแบบ "X ago"
  • ลบการขึ้นบรรทัดใหม่ซ้อนกันในรายการ (list items) ออกแล้ว เพื่อไม่ให้การคัดลอกคำตอบไปวางใน Word หรือ Google Docs กลายเป็นรายการที่เว้นบรรทัดคู่

ความน่าเชื่อถือ

ความครอบคลุมของการทดสอบ (test coverage) สำหรับ consumer helper ที่ได้รับผลกระทบเพิ่มขึ้นจาก 40% เป็น 73% (19 → 57 การทดสอบ) พร้อมการทดสอบใหม่สำหรับการตรวจจับการค้าง การป้องกันคำขอพร้อมกัน และกรณีขอบเขต (edge case) ของการยกเลิก

การจัดการข้อมูลยังคงบังคับใช้เหมือนเดิม

เป็นเพียงการแก้ไขด้านความน่าเชื่อถือและจังหวะเวลาเท่านั้น — ไม่มีการเปลี่ยนแปลงข้อมูลที่ tool call ใด ๆ สามารถเข้าถึงได้ ดู Data Security

ပယ်ဖျက်မှုနှင့် Streaming ယုံကြည်စိတ်ချရမှု

တစ်ချိန်တည်း တောင်းဆိုမှုများ (concurrent requests)၊ ရပ်တန့်နေသော တုံ့ပြန်မှုများ (stale responses) နှင့် ပယ်ဖျက်ခြင်း race condition များကို ဖြေရှင်းပေးသော ယုံကြည်စိတ်ချရမှု ပြုပြင်မှု — stop ခလုတ်အသစ်၊ ရှည်လျားသော streaming timeout များ နှင့် Brave News ရှာဖွေရေးကိရိယာ၏ ရက်စွဲစစ်ထုတ်မှု ပြင်ဆင်ချက်တို့ ပါဝင်သည်

Concurrency နှင့် Race Condition များ

  • တစ်ချိန်တည်း တုံ့ပြန်မှု ကာကွယ်ခြင်း - session တစ်ခုတည်းအတွက် တုံ့ပြန်မှု streaming နေဆဲအချိန်တွင် စာအသစ်ကို ပယ်ချမည်၊ atomic database lock များဖြင့် အတည်ပြု အကောင်အထည်ဖော်ထားသည်
  • ရပ်တန့်နေသော တုံ့ပြန်မှု ရှာဖွေတွေ့ရှိခြင်း - 10 မိနစ်ထက် ကြာသော tracker များကို အလိုအလျောက် failed ဟု အမှတ်အသားပြုလိုက်သည်၊ တောင်းဆိုမှုအသစ်များကို ပိတ်ဆို့မှု မဖြစ်စေရန်
  • ပယ်ဖျက်ခြင်း တိုးတက်မှုများ - tracker မတည်ရှိမီ စောစီးစွာ ပယ်ဖျက်ခြင်းအတွက် flag များနှင့် ပယ်ဖျက်မှု ထပ်ခါထပ်ခါ မဖြစ်စေရေး ကာကွယ်မှု၊ ပယ်ဖျက်ရန် ကြိုးစားမီ tracker ၏ terminal status ကို စစ်ဆေးသည်

Streaming နှင့် Timeout များ

ချိန်ညှိချက်ယခင်ပြင်ဆင်ပြီးနောက်
Celery soft limit5 မိနစ်12 မိနစ်
Celery hard limit6 မိနစ်15 မိနစ်
Redis buffer TTL5 မိနစ်15 မိနစ်

stop ခလုတ်အသစ်က တုံ့ပြန်မှု ထုတ်လုပ်နေစဉ် အလယ်တွင် ပယ်ဖျက်နိုင်စေပြီး ရပ်တန့်နေသော တုံ့ပြန်မှုများ (15 မိနစ်အတွင်း update မရှိသော) ယခုအခါ အကန့်အသတ်မရှိစောင့်နေမည့်အစား အလိုအလျောက် ပြန်လည်ရယူသည်

အခြား ပြင်ဆင်ချက်များ

  • Brave News search သည် API ၏ freshness parameter ကို မှန်ကန်စွာ အသုံးပြုပြီ၊ preset code များနှင့် "X ago" ပုံစံ ယခင်ရှာဖွေမှုများကို ပံ့ပိုးသည်
  • list item များတွင် ထပ်နေသော line break များကို ဖယ်ရှားလိုက်သည်၊ တုံ့ပြန်မှုကို Word သို့မဟုတ် Google Docs သို့ ကူးထည့်သည့်အခါ list များ line နှစ်ဆ space မဖြစ်စေတော့ပါ

ယုံကြည်စိတ်ချရမှု

ထိခိုက်သော consumer helper များအတွက် test coverage သည် 40% မှ 73% သို့ (test 19 → 57) တိုးတက်လာပြီး ရပ်တန့်မှု ရှာဖွေခြင်း၊ တစ်ချိန်တည်း တောင်းဆိုမှု ကာကွယ်ခြင်း နှင့် ပယ်ဖျက်ခြင်း edge case များအတွက် test အသစ်များ ထည့်သွင်းထားသည်

ဒေတာ ကိုင်တွယ်မှု စည်းမျဉ်းများ ဆက်လက် သက်ရောက်သည်

ဤသည်မှာ ယုံကြည်စိတ်ချရမှုနှင့် အချိန်ဆိုင်ရာ ပြင်ဆင်ချက်သာဖြစ်သည် — tool call မည်သည့်ဒေတာကို ဝင်ရောက်ရယူနိုင်သည်ဆိုသည်မှာ ပြောင်းလဲခြင်း မရှိပါ။ Data Security ကို ကြည့်ပါ
