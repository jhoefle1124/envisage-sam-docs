Administrative Management of Prompt Writing

This is the backend architecture behind the prompt template system — the data model, permissions, and API surface that Prompt Studio and Prompt Templates are built on top of. If you're an admin or developer who needs to understand how templates are structured and governed, this page is for you.

How This Fits Together

Three pages on this site now cover the prompt template system from different angles:

Prompt Templates

The user-facing guide: what templates are, placeholder syntax, and example templates. See Prompt Templates.

Prompt Studio

The admin editing UI: hierarchical categories, the markdown editor, drag-and-drop ordering. See Prompt Studio.

This page

The underlying data model, access control, and API endpoints that make the other two possible.

The Data Model

ModelKey Fields
PromptTemplateCategoryname (unique), description, slug (unique), order, is_active
PromptTemplatename, description, category (linked to a category), template_text, placeholders (auto-detected list), is_active, created_by, order

Categories protect against accidental data loss: a category can't be deleted while it still has templates attached to it. Templates support soft deletion rather than being permanently removed.

Access Control

Every admin view for categories and templates — list, create, edit, delete, and detail/preview — requires site-admin access. Regular users never see the admin interface; they only see templates once an admin has published them as active.

Placeholder Detection & Validation

Placeholders are detected automatically from the template text using the pattern {{...}}, validated for correct syntax, and stored alongside the template for quick reference — matching is case-insensitive, so {{Company}} and {{company}} are treated as equivalent. The four placeholders SAM understands are documented on the Prompt Templates page.

API Surface

EndpointPurpose
Templates list APIReturns all active templates, grouped by category, for use in the chat interface
Template detail APIReturns the full detail of a single template by ID

Both endpoints require a logged-in user. This is the surface the chat interface's template picker is expected to call — see Future Considerations below for what's not built yet.

Reliability

Tested across the stack. Coverage spans model validation and placeholder extraction, view-level permission checks, API response correctness, and form-level placeholder validation, including a range of placeholder-format edge cases.

What's Not Built Yet

Frontend selection is a separate, later piece of work. The in-chat modal for picking a template, company autocomplete for filling in placeholders, real-time preview with replacements, template search, and multi-company placeholder support are planned as follow-on frontend work — not part of this backend system. Don't assume the full in-chat experience is live; check with your team lead on current availability.

Data Handling Still Applies

Templates stored through this system are visible to every user once published. The same guidance from Prompt Studio applies here: template structure and placeholders are Tier 1 boilerplate meant for reuse — don't write a real client name, engagement detail, or case specific directly into the stored template text, per Data Security.

การบริหารจัดการงานเขียนพรอมต์สำหรับผู้ดูแลระบบ

นี่คือสถาปัตยกรรมฝั่งแบ็กเอนด์ (backend) ที่อยู่เบื้องหลังระบบเทมเพลตพรอมต์ — โมเดลข้อมูล การกำหนดสิทธิ์การเข้าถึง และขอบเขตของ API ที่หน้า Prompt Studio และ Prompt Templates ถูกสร้างขึ้นบนพื้นฐานนี้ หากคุณเป็นผู้ดูแลระบบหรือนักพัฒนาที่ต้องการเข้าใจว่าเทมเพลตถูกจัดโครงสร้างและควบคุมดูแลอย่างไร หน้านี้เหมาะสำหรับคุณ

ภาพรวมความเชื่อมโยงของแต่ละหน้า

ขณะนี้มีสามหน้าบนเว็บไซต์นี้ที่ครอบคลุมระบบเทมเพลตพรอมต์จากมุมมองที่แตกต่างกัน:

Prompt Templates

คู่มือสำหรับผู้ใช้งาน: เทมเพลตคืออะไร ไวยากรณ์ของ placeholder และตัวอย่างเทมเพลต ดูที่ Prompt Templates

Prompt Studio

UI สำหรับผู้ดูแลระบบในการแก้ไข: หมวดหมู่แบบลำดับชั้น ตัวแก้ไข markdown และการจัดลำดับด้วยการลากวาง ดูที่ Prompt Studio

หน้านี้

โมเดลข้อมูลพื้นฐาน การควบคุมการเข้าถึง และ API endpoint ที่ทำให้อีกสองหน้าเป็นไปได้

โมเดลข้อมูล

โมเดลฟิลด์สำคัญ
PromptTemplateCategoryname (ไม่ซ้ำกัน), description, slug (ไม่ซ้ำกัน), order, is_active
PromptTemplatename, description, category (เชื่อมโยงกับหมวดหมู่หนึ่ง), template_text, placeholders (รายการที่ตรวจพบอัตโนมัติ), is_active, created_by, order

หมวดหมู่มีกลไกป้องกันข้อมูลสูญหายโดยไม่ตั้งใจ: หมวดหมู่จะไม่สามารถลบได้ตราบใดที่ยังมีเทมเพลตผูกอยู่กับหมวดหมู่นั้น ส่วนเทมเพลตรองรับการลบแบบ soft deletion แทนที่จะถูกลบออกอย่างถาวร

การควบคุมการเข้าถึง

ทุกหน้าจอผู้ดูแลระบบสำหรับหมวดหมู่และเทมเพลต — รายการ การสร้าง การแก้ไข การลบ และรายละเอียด/ตัวอย่าง — ต้องมีสิทธิ์ site-admin ผู้ใช้งานทั่วไปจะไม่เห็นอินเทอร์เฟซของผู้ดูแลระบบเลย พวกเขาจะเห็นเทมเพลตก็ต่อเมื่อผู้ดูแลระบบเผยแพร่เทมเพลตนั้นเป็น active แล้วเท่านั้น

การตรวจจับและตรวจสอบความถูกต้องของ Placeholder

Placeholder จะถูกตรวจจับโดยอัตโนมัติจากข้อความเทมเพลตโดยใช้รูปแบบ {{...}} ตรวจสอบความถูกต้องของไวยากรณ์ และจัดเก็บไว้คู่กับเทมเพลตเพื่อให้อ้างอิงได้อย่างรวดเร็ว — การจับคู่ไม่สนใจตัวพิมพ์เล็ก-ใหญ่ ดังนั้น {{Company}} และ {{company}} จะถือว่าเหมือนกัน placeholder ทั้งสี่แบบที่ SAM รองรับมีอธิบายไว้ในหน้า Prompt Templates

ขอบเขตของ API

Endpointวัตถุประสงค์
Templates list APIส่งคืนเทมเพลตที่ active ทั้งหมด จัดกลุ่มตามหมวดหมู่ เพื่อใช้งานในอินเทอร์เฟซแชท
Template detail APIส่งคืนรายละเอียดทั้งหมดของเทมเพลตหนึ่งรายการตาม ID

ทั้งสอง endpoint ต้องการผู้ใช้ที่ล็อกอินแล้ว นี่คือขอบเขตที่ตัวเลือกเทมเพลตของอินเทอร์เฟซแชทคาดว่าจะเรียกใช้ — ดูหัวข้อสิ่งที่ยังไม่ได้สร้างด้านล่างสำหรับสิ่งที่ยังไม่มี

ความน่าเชื่อถือ

ทดสอบครอบคลุมทุกชั้นของระบบ การทดสอบครอบคลุมการตรวจสอบโมเดลและการดึง placeholder การตรวจสอบสิทธิ์ในระดับ view ความถูกต้องของการตอบกลับจาก API และการตรวจสอบ placeholder ในระดับฟอร์ม รวมถึงกรณีขอบเขตต่าง ๆ ของรูปแบบ placeholder

สิ่งที่ยังไม่ได้สร้าง

การเลือกเทมเพลตฝั่ง frontend เป็นงานแยกต่างหากที่จะทำในภายหลัง โมดัลสำหรับเลือกเทมเพลตในหน้าแชท การเติมชื่อบริษัทอัตโนมัติ (autocomplete) สำหรับกรอก placeholder การแสดงตัวอย่างแบบเรียลไทม์พร้อมค่าที่แทนที่ การค้นหาเทมเพลต และการรองรับ placeholder หลายบริษัท ล้วนวางแผนไว้เป็นงาน frontend ที่จะตามมา — ไม่ใช่ส่วนหนึ่งของระบบแบ็กเอนด์นี้ อย่าคาดเดาว่าประสบการณ์การใช้งานในหน้าแชทแบบเต็มรูปแบบพร้อมใช้งานแล้ว โปรดตรวจสอบกับหัวหน้าทีมของคุณถึงความพร้อมใช้งานในปัจจุบัน

กฎการจัดการข้อมูลยังคงมีผลบังคับใช้

เทมเพลตที่จัดเก็บผ่านระบบนี้จะปรากฏต่อผู้ใช้งานทุกคนเมื่อเผยแพร่แล้ว คำแนะนำเดียวกันจาก Prompt Studio ใช้ได้กับหน้านี้เช่นกัน: โครงสร้างเทมเพลตและ placeholder เป็น boilerplate ระดับ Tier 1 ที่มีไว้สำหรับใช้ซ้ำ — อย่าเขียนชื่อลูกค้าจริง รายละเอียดงาน (engagement) หรือรายละเอียดเฉพาะเคสลงในข้อความเทมเพลตที่จัดเก็บไว้โดยตรง ตามที่ระบุใน Data Security

SAM တွင် အသစ် · နည်းပညာဆိုင်ရာ ကိုးကားချက်

Prompt ရေးသားမှုအတွက် Admin စီမံခန့်ခွဲမှု

ဤစာမျက်နှာသည် Prompt Template စနစ်၏ နောက်ကွယ်ရှိ backend architecture ဖြစ်သည် — Prompt Studio နှင့် Prompt Templates တို့ တည်ဆောက်ရာတွင် အခြေခံအဖြစ် အသုံးပြုသည့် data model, ခွင့်ပြုချက်များ နှင့် API surface တို့ဖြစ်သည်။ Template များကို မည်သို့ဖွဲ့စည်းပြီး ထိန်းချုပ်ထားသည်ကို နားလည်လိုသော admin သို့မဟုတ် developer တစ်ဦးဖြစ်ပါက ဤစာမျက်နှာသည် သင့်အတွက်ဖြစ်သည်။

ဤစာမျက်နှာများ မည်သို့ ဆက်စပ်နေသည်

ယခုအခါ ဤဝဘ်ဆိုက်ရှိ စာမျက်နှာသုံးခုသည် Prompt Template စနစ်ကို ရှုထောင့်အမျိုးမျိုးမှ ဖော်ပြထားသည် —

Prompt Templates

အသုံးပြုသူများအတွက် လမ်းညွှန်: Template ဆိုသည်မှာ ဘာလဲ၊ placeholder syntax နှင့် နမူနာ Template များ။ Prompt Templates တွင် ကြည့်ပါ။

Prompt Studio

Admin တည်းဖြတ်သည့် UI: အဆင့်ဆင့် category များ၊ markdown editor၊ drag-and-drop စီစဉ်မှု။ Prompt Studio တွင် ကြည့်ပါ။

ဤစာမျက်နှာ

အခြားစာမျက်နှာနှစ်ခုကို ဖြစ်နိုင်စေသော အခြေခံ data model၊ access control နှင့် API endpoint များ

Data Model

Modelအဓိက Field များ
PromptTemplateCategoryname (ထူးခြားသော), description, slug (ထူးခြားသော), order, is_active
PromptTemplatename, description, category (category တစ်ခုနှင့် ချိတ်ဆက်ထား), template_text, placeholders (အလိုအလျောက် ရှာဖွေတွေ့ရှိသော list), is_active, created_by, order

Category များသည် မတော်တဆ ဒေတာဆုံးရှုံးမှုမှ ကာကွယ်ပေးသည်: category တစ်ခုတွင် Template များ ဆက်စပ်နေသေးလျှင် ထို category ကို ဖျက်၍မရပါ။ Template များသည် အပြီးတိုင် ဖယ်ရှားခြင်းထက် soft deletion ကို ပံ့ပိုးထားသည်။

Access Control

Category နှင့် Template များအတွက် admin view အားလုံး — list, create, edit, delete နှင့် detail/preview — သည် site-admin ခွင့်ပြုချက် လိုအပ်သည်။ ပုံမှန်အသုံးပြုသူများသည် admin interface ကို လုံးဝမမြင်ရပါ; admin မှ active အဖြစ် ထုတ်ပြန်ပြီးမှသာ Template များကို မြင်ရသည်။

Placeholder ရှာဖွေခြင်းနှင့် စစ်ဆေးခြင်း

Placeholder များကို {{...}} ပုံစံသုံး၍ Template စာသားမှ အလိုအလျောက် ရှာဖွေတွေ့ရှိပြီး syntax မှန်ကန်မှုကို စစ်ဆေးကာ Template နှင့်အတူ လျင်မြန်စွာ ကိုးကားနိုင်ရန် သိမ်းဆည်းထားသည် — ကိုက်ညီမှုစစ်ဆေးရာတွင် စာလုံးအကြီးအသေး ခွဲခြားမထားပါ၊ ထို့ကြောင့် {{Company}} နှင့် {{company}} ကို တူညီသည်ဟု ယူဆသည်။ SAM နားလည်သော placeholder လေးမျိုးကို Prompt Templates စာမျက်နှာတွင် မှတ်တမ်းတင်ထားသည်။

API Surface

Endpointရည်ရွယ်ချက်
Templates list APIactive Template အားလုံးကို category အလိုက် စုစည်းပြီး chat interface တွင် အသုံးပြုရန် ပြန်ပေးသည်
Template detail APITemplate တစ်ခု၏ အသေးစိတ်အချက်အလက်အားလုံးကို ID ဖြင့် ပြန်ပေးသည်

Endpoint နှစ်ခုစလုံးသည် login ဝင်ထားသော အသုံးပြုသူ လိုအပ်သည်။ ဤသည်မှာ chat interface ၏ Template ရွေးချယ်ရေး UI မှ ခေါ်သုံးရန် မျှော်လင့်ထားသော surface ဖြစ်သည် — ဘာများ မတည်ဆောက်ရသေးသည်ကို အောက်ပါ Future Considerations တွင် ကြည့်ပါ။

ယုံကြည်စိတ်ချရမှု

Stack တစ်ခုလုံးတွင် စမ်းသပ်ထားသည်။ Coverage သည် model validation နှင့် placeholder ထုတ်ယူခြင်း၊ view level ခွင့်ပြုချက်စစ်ဆေးမှု၊ API response မှန်ကန်မှု နှင့် form level placeholder validation အထိ ပါဝင်ပြီး placeholder ပုံစံ အမျိုးမျိုး၏ edge case များပါ ပါဝင်သည်။

မတည်ဆောက်ရသေးသော အရာများ

Frontend ရွေးချယ်မှုသည် နောက်မှ လုပ်ဆောင်မည့် သီးခြားအလုပ်တစ်ခု ဖြစ်သည်။ Template ရွေးချယ်ရန် chat တွင်း modal, placeholder ဖြည့်ရန် company autocomplete, အစားထိုးချက်များပါသော real-time preview, Template ရှာဖွေခြင်း နှင့် company အများအပြားအတွက် placeholder ပံ့ပိုးမှု တို့ကို နောက်ဆက်တွဲ frontend အလုပ်အဖြစ် စီစဉ်ထားသည် — ဤ backend စနစ်၏ အစိတ်အပိုင်းမဟုတ်ပါ။ chat တွင်း အပြည့်အဝ အတွေ့အကြုံသည် အသုံးပြုနိုင်ပြီဟု မယူဆပါနှင့်; လက်ရှိ availability ကို သင့်အဖွဲ့ခေါင်းဆောင်နှင့် စစ်ဆေးပါ။

ဒေတာကိုင်တွယ်မှု စည်းမျဉ်းများ ဆက်လက် သက်ရောက်နေဆဲ

ဤစနစ်မှတစ်ဆင့် သိမ်းဆည်းထားသော Template များကို ထုတ်ပြန်ပြီးသည်နှင့် အသုံးပြုသူတိုင်း မြင်ရသည်။ Prompt Studio မှ လမ်းညွှန်ချက်အတိုင်းပင် ဤနေရာတွင်လည်း သက်ရောက်သည်: Template ဖွဲ့စည်းပုံနှင့် placeholder များသည် ပြန်လည်အသုံးပြုရန်အတွက် Tier 1 boilerplate ဖြစ်သည် — Data Security အရ တကယ့်ဖောက်သည်နာမည်၊ engagement အသေးစိတ် သို့မဟုတ် ကိစ္စတစ်ခုချင်းစီ၏ အသေးစိတ်ကို သိမ်းဆည်းထားသော Template စာသားထဲသို့ တိုက်ရိုက် မရေးပါနှင့်။
