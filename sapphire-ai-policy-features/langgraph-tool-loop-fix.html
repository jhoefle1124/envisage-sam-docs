LangGraph Tool Loop Fix

A fix for a class of bugs where the chatbot could get stuck repeatedly calling tools without ever returning an answer — corrected through proper LangGraph message handling and a smarter recursion-limit mechanism.

What Was Wrong

  • Tool calling loops: improper message flow between LLM calls and tool executions could cause the chatbot to call tools indefinitely instead of returning a response.
  • Anthropic provider errors: an unsupported request_timeout parameter was causing initialization failures for Claude specifically.

The Fix

  • Added a proper messages list to the graph state, using LangGraph-compliant message types (HumanMessage, AIMessage, ToolMessage, SystemMessage) throughout.
  • Replaced a hardcoded max-tool-executions setting with LangGraph's built-in RemainingSteps mechanism, which degrades gracefully with a user-friendly message instead of an abrupt error as a conversation approaches its depth limit.
  • Corrected the Anthropic provider to use the timeout parameter it actually supports, while keeping other providers (OpenAI, Fireworks) unaffected.

Reliability

All existing tests pass, plus new coverage for message handling and remaining-steps behavior, verified with manual testing across tool-calling scenarios.

Data Handling Still Applies

This is a reliability fix to how the chatbot orchestrates tool calls — it does not change what data any tool is allowed to touch. Standard Data Security rules continue to apply to every tool call.

การแก้ไขปัญหา Tool Loop ใน LangGraph

การแก้ไขบั๊กประเภทหนึ่งที่ทำให้ chatbot อาจติดค้างอยู่ในสถานะเรียกใช้ tools ซ้ำ ๆ โดยไม่เคยส่งคำตอบกลับมา — แก้ไขผ่านการจัดการ message ที่ถูกต้องใน LangGraph และกลไกจำกัด recursion ที่ฉลาดขึ้น

สิ่งที่ผิดพลาด

  • Tool calling loops: การไหลของ message ที่ไม่ถูกต้องระหว่างการเรียก LLM และการทำงานของ tools อาจทำให้ chatbot เรียกใช้ tools ซ้ำไปเรื่อย ๆ อย่างไม่มีที่สิ้นสุด แทนที่จะส่งคำตอบกลับมา
  • Anthropic provider errors: พารามิเตอร์ request_timeout ที่ไม่รองรับ ทำให้เกิดความล้มเหลวในการ initialize โดยเฉพาะกับ Claude

การแก้ไข

  • เพิ่ม messages list ที่ถูกต้องเข้าไปใน graph state โดยใช้ message types ที่สอดคล้องกับ LangGraph (HumanMessage, AIMessage, ToolMessage, SystemMessage) ตลอดทั้งระบบ
  • แทนที่การตั้งค่า max-tool-executions แบบ hardcode ด้วยกลไก RemainingSteps ที่มีอยู่ในตัวของ LangGraph ซึ่งจะลดระดับการทำงานอย่างนุ่มนวลพร้อมข้อความที่เป็นมิตรต่อผู้ใช้ แทนที่จะแสดง error กะทันหันเมื่อบทสนทนาใกล้ถึงขีดจำกัดความลึก
  • แก้ไข Anthropic provider ให้ใช้พารามิเตอร์ timeout ที่รองรับจริง โดยไม่กระทบต่อ providers อื่น ๆ (OpenAI, Fireworks)

ความน่าเชื่อถือ

การทดสอบเดิมทั้งหมดผ่านเรียบร้อย พร้อมเพิ่ม coverage ใหม่ สำหรับการจัดการ message และพฤติกรรมของ remaining-steps ตรวจสอบเพิ่มเติมด้วยการทดสอบด้วยตนเองในสถานการณ์การเรียกใช้ tools ต่าง ๆ

การจัดการข้อมูลยังคงบังคับใช้เช่นเดิม

นี่คือการแก้ไขด้านความน่าเชื่อถือในวิธีที่ chatbot จัดการ orchestration การเรียกใช้ tools เท่านั้น — ไม่ได้เปลี่ยนแปลงว่าข้อมูลใดที่ tool แต่ละตัวได้รับอนุญาตให้เข้าถึง กฎ Data Security มาตรฐานยังคงบังคับใช้กับการเรียกใช้ tool ทุกครั้ง

LangGraph Tool Loop ပြဿနာ ပြင်ဆင်ချက်

chatbot သည် အဖြေတစ်ခုမျှ မပြန်ပေးဘဲ tools များကို ထပ်ခါထပ်ခါ ခေါ်နေရာတွင် ပိတ်မိသွားနိုင်သည့် bug အမျိုးအစားတစ်ခုအတွက် ပြင်ဆင်ချက် — LangGraph ၏ message handling ကို မှန်ကန်စွာ ပြုလုပ်ခြင်းနှင့် ပိုမိုပါးနပ်သော recursion-limit ယန္တရားဖြင့် ပြင်ဆင်ထားသည်။

ဘာမှားယွင်းခဲ့သနည်း

  • Tool calling loops: LLM ခေါ်ဆိုမှုများနှင့် tool execution များကြား message flow မမှန်ကန်ခြင်းကြောင့် chatbot သည် အဖြေပြန်မပေးဘဲ tools များကို အဆုံးမရှိ ခေါ်ဆိုနေနိုင်သည်။
  • Anthropic provider errors: ပံ့ပိုးမထားသော request_timeout parameter သည် Claude အတွက်သီးသန့် initialization ပျက်ကွက်မှုများ ဖြစ်စေခဲ့သည်။

ပြင်ဆင်ချက်

  • graph state ထဲသို့ မှန်ကန်သော messages list ကို ထည့်သွင်းပြီး၊ LangGraph နှင့် ကိုက်ညီသော message types (HumanMessage, AIMessage, ToolMessage, SystemMessage) များကို တစ်ချောင်းလုံး အသုံးပြုထားသည်။
  • hardcode လုပ်ထားသော max-tool-executions setting ကို LangGraph ၏ built-in RemainingSteps ယန္တရားဖြင့် အစားထိုးထားသည်။ ၎င်းသည် စကားဝိုင်း depth limit သို့ ရောက်ရှိလာသောအခါ ရုတ်တရက် error မဖြစ်စေဘဲ အသုံးပြုသူ နားလည်လွယ်သော message ဖြင့် နူးညံ့စွာ degrade ဖြစ်စေသည်။
  • Anthropic provider ကို ၎င်းအမှန်တကယ် ပံ့ပိုးထားသော timeout parameter ကို အသုံးပြုစေရန် ပြင်ဆင်ထားပြီး၊ အခြား providers (OpenAI, Fireworks) များကို ထိခိုက်မှု မရှိစေပါ။

ယုံကြည်စိတ်ချရမှု

ရှိပြီးသား test များအားလုံး အောင်မြင်ပြီး၊ message handling နှင့် remaining-steps အပြုအမူအတွက် coverage အသစ်များပါ ထပ်ထည့်ထားသည် — tool-calling scenarios အမျိုးမျိုးတွင် manual testing ဖြင့် အတည်ပြုထားသည်။

ဒေတာကိုင်တွယ်မှု စည်းမျဉ်းများ ဆက်လက်သက်ရောက်

ဤသည်မှာ chatbot က tool calls များကို orchestrate ပြုလုပ်ပုံအတွက် ယုံကြည်စိတ်ချရမှု ပြင်ဆင်ချက်တစ်ခုသာ ဖြစ်ပြီး — tool တစ်ခုချင်းစီ ထိတွေ့ခွင့်ရှိသော ဒေတာကို ပြောင်းလဲခြင်း မဟုတ်ပါ။ standard Data Security စည်းမျဉ်းများသည် tool call တိုင်းတွင် ဆက်လက် သက်ရောက်နေမည်ဖြစ်သည်။
