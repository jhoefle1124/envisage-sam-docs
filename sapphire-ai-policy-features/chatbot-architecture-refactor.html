Chatbot Architecture Refactor

A ground-up architectural pass on the chatbot backend: a message handler registry for WebSocket traffic, a proper lifecycle manager for LangGraph, a consolidated document manager, and a real exception hierarchy — plus meaningful performance and file-handling gains.

What's New

  • MessageHandlerRegistry: a registry pattern for WebSocket message handling that replaces a sprawling consumer with focused, testable handlers.
  • GraphManager: a proper lifecycle manager for LangGraph instances following async application best practices.
  • DocumentManager: all document-related operations consolidated into one dedicated manager class.
  • Exception hierarchy: 15+ specific exception types replace generic error handling for precise debugging.

Performance Gains

AreaImprovement
Database queries+30-50% faster for user/session lookups via indexes and select_related/prefetch_related
Large file processing+60% memory efficiency, via lazy loading for PDFs over 2MB
Embedding generation+25% throughput — batch size increased from 50 to 100
WebSocket handling+40% reduced complexity from the registry pattern

Enhanced Document Processing

  • Added support for DOCX and XLSX file formats, on top of existing PDF handling.
  • File size limits raised: default files 10MB → 25MB, Excel files up to 50MB.
  • Real-time progress tracking for uploads and processing, with async handling for large files.
  • Large PDFs (100+ pages) use batch processing (50 pages at a time); documents cap at 500 chunks to prevent runaway processing.

Reliability

Backward compatible, additive migrations only. 70 files changed with 20+ new test files. No configuration changes required — existing documents automatically pick up the new progress tracking.

Data Handling Still Applies

This is an infrastructure refactor — it changes how documents and messages move through the system, not what tier they carry. The Data Security rules on what may be uploaded still apply exactly as before.

ใหม่ใน SAM · เอกสารอ้างอิงทางเทคนิค

การปรับสถาปัตยกรรมของแชทบอทครั้งใหญ่

การปรับปรุงสถาปัตยกรรมของแบ็กเอนด์แชทบอทตั้งแต่รากฐาน: registry สำหรับจัดการ handler ของข้อความ WebSocket ตัวจัดการวงจรชีวิต (lifecycle) ที่เหมาะสมสำหรับ LangGraph ตัวจัดการเอกสารแบบรวมศูนย์ และลำดับชั้น exception ที่แท้จริง — พร้อมกับการปรับปรุงประสิทธิภาพและการจัดการไฟล์ที่เห็นผลชัดเจน

สิ่งที่ใหม่

  • MessageHandlerRegistry: รูปแบบ registry สำหรับจัดการข้อความ WebSocket ที่เข้ามาแทนที่ consumer ขนาดใหญ่เพียงตัวเดียวด้วย handler ที่แยกหน้าที่ชัดเจนและทดสอบได้
  • GraphManager: ตัวจัดการวงจรชีวิตที่เหมาะสมสำหรับ instance ของ LangGraph ตามแนวปฏิบัติที่ดีที่สุดของแอปพลิเคชันแบบ async
  • DocumentManager: การทำงานที่เกี่ยวข้องกับเอกสารทั้งหมดถูกรวมไว้ในคลาสตัวจัดการเฉพาะเพียงตัวเดียว
  • ลำดับชั้น Exception: exception ชนิดเฉพาะกว่า 15 แบบเข้ามาแทนที่การจัดการข้อผิดพลาดทั่วไป เพื่อการดีบักที่แม่นยำ

ผลด้านประสิทธิภาพ

ด้านการปรับปรุง
การ query ฐานข้อมูลเร็วขึ้น 30-50% สำหรับการค้นหาผู้ใช้/เซสชัน ผ่านการทำ index และ select_related/prefetch_related
การประมวลผลไฟล์ขนาดใหญ่ประสิทธิภาพหน่วยความจำเพิ่มขึ้น 60% ผ่านการทำ lazy loading สำหรับ PDF ที่มีขนาดเกิน 2MB
การสร้าง embeddingปริมาณงานเพิ่มขึ้น 25% — เพิ่มขนาด batch จาก 50 เป็น 100
การจัดการ WebSocketลดความซับซ้อนลง 40% จากรูปแบบ registry

การประมวลผลเอกสารที่ปรับปรุงแล้ว

  • เพิ่มการรองรับไฟล์ประเภท DOCX และ XLSX นอกเหนือจากการรองรับ PDF ที่มีอยู่เดิม
  • ขยายขีดจำกัดขนาดไฟล์: ไฟล์ทั่วไปจาก 10MB เป็น 25MB ไฟล์ Excel สูงสุด 50MB
  • ติดตามความคืบหน้าแบบเรียลไทม์สำหรับการอัปโหลดและการประมวลผล พร้อมการจัดการแบบ async สำหรับไฟล์ขนาดใหญ่
  • PDF ขนาดใหญ่ (100 หน้าขึ้นไป) ใช้การประมวลผลแบบ batch (ครั้งละ 50 หน้า); เอกสารจำกัดไว้ที่ 500 chunk เพื่อป้องกันการประมวลผลที่ไม่มีที่สิ้นสุด

ความน่าเชื่อถือ

เข้ากันได้กับระบบเดิม เป็นเพียงการ migration แบบเพิ่มเติมเท่านั้น เปลี่ยนแปลง 70 ไฟล์ พร้อมไฟล์ทดสอบใหม่กว่า 20 ไฟล์ ไม่จำเป็นต้องเปลี่ยนแปลงการตั้งค่าใด ๆ — เอกสารที่มีอยู่เดิมจะได้รับการติดตามความคืบหน้าแบบใหม่โดยอัตโนมัติ

กฎการจัดการข้อมูลยังคงมีผลบังคับใช้

นี่คือการปรับโครงสร้างพื้นฐาน — เปลี่ยนแปลงวิธีที่เอกสารและข้อความเคลื่อนที่ผ่านระบบ ไม่ได้เปลี่ยนแปลงระดับชั้นข้อมูลที่เอกสารเหล่านั้นถืออยู่ กฎของ Data Security เกี่ยวกับสิ่งที่สามารถอัปโหลดได้ยังคงมีผลบังคับใช้เหมือนเดิมทุกประการ

SAM တွင် အသစ် · နည်းပညာဆိုင်ရာ ကိုးကားချက်

Chatbot Architecture ပြန်လည်တည်ဆောက်မှု

Chatbot backend ကို အခြေခံမှ ပြန်လည်ဒီဇိုင်းဆွဲထားခြင်း — WebSocket traffic အတွက် message handler registry, LangGraph အတွက် သင့်လျော်သော lifecycle manager, စုစည်းထားသော document manager နှင့် တကယ့် exception hierarchy — ထို့အပြင် သိသာထင်ရှားသော ဆောင်ရွက်နိုင်စွမ်းနှင့် ဖိုင်ကိုင်တွယ်မှု တိုးတက်မှုများပါ ပါဝင်သည်။

အသစ်ပါဝင်သည့်အရာများ

  • MessageHandlerRegistry: ကျယ်ပြန့်နေသော consumer တစ်ခုကို အာရုံစိုက်ပြီး စမ်းသပ်နိုင်သော handler များဖြင့် အစားထိုးသည့် WebSocket message handling အတွက် registry pattern
  • GraphManager: async application အကောင်းဆုံးအလေ့အကျင့်များအတိုင်း LangGraph instance များအတွက် သင့်လျော်သော lifecycle manager
  • DocumentManager: document ဆိုင်ရာ လုပ်ဆောင်ချက်အားလုံးကို သီးသန့် manager class တစ်ခုတည်းထဲသို့ စုစည်းထားသည်
  • Exception hierarchy: တိကျသော debug လုပ်နိုင်ရန် ယေဘုယျ error handling ကို အစားထိုးသည့် သီးခြား exception အမျိုးအစား ၁၅ မျိုးကျော်

ဆောင်ရွက်နိုင်စွမ်း တိုးတက်မှုများ

နယ်ပယ်တိုးတက်မှု
Database query များindex များနှင့် select_related/prefetch_related မှတစ်ဆင့် user/session ရှာဖွေမှုအတွက် +30-50% ပိုမြန်
ဖိုင်ကြီးများ ပြုပြင်ခြင်း2MB ကျော် PDF များအတွက် lazy loading အသုံးပြုခြင်းဖြင့် memory efficiency +60%
Embedding ထုတ်လုပ်ခြင်းthroughput +25% — batch size ကို 50 မှ 100 သို့ တိုးမြှင့်
WebSocket ကိုင်တွယ်ခြင်းregistry pattern ကြောင့် ရှုပ်ထွေးမှု -40%

တိုးတက်ထားသော Document ကိုင်တွယ်ခြင်း

  • ရှိပြီးသား PDF ကိုင်တွယ်မှုအပြင် DOCX နှင့် XLSX ဖိုင်ပုံစံများအတွက် ပံ့ပိုးမှု ထပ်ထည့်ထားသည်
  • ဖိုင်အရွယ်အစား ကန့်သတ်ချက် တိုးမြှင့်: ပုံမှန်ဖိုင် 10MB → 25MB, Excel ဖိုင် 50MB အထိ
  • Upload နှင့် processing အတွက် real-time progress tracking၊ ဖိုင်ကြီးများအတွက် async ကိုင်တွယ်မှုနှင့်အတူ
  • PDF ကြီးများ (စာမျက်နှာ 100+) သည် batch processing (တစ်ကြိမ်လျှင် 50 မျက်နှာ) ကို အသုံးပြုသည်; processing မထိန်းချုပ်နိုင်အောင် ဖြစ်မသွားစေရန် document တစ်ခုကို chunk 500 ဖြင့် ကန့်သတ်ထားသည်

ယုံကြည်စိတ်ချရမှု

ရှေ့နောက် လိုက်ဖက်ညီပြီး ထပ်ဆင့်ထည့်သော migration များသာ ဖြစ်သည်။ Test ဖိုင်အသစ် 20+ ဖိုင်ဖြင့် ဖိုင် 70 ခု ပြောင်းလဲထားသည်။ configuration ပြောင်းလဲရန် မလိုအပ်ပါ — ရှိပြီးသား document များသည် progress tracking အသစ်ကို အလိုအလျောက် ရရှိသွားမည်။

ဒေတာကိုင်တွယ်မှု စည်းမျဉ်းများ ဆက်လက် သက်ရောက်နေဆဲ

ဤသည်မှာ document နှင့် message များ စနစ်ထဲကို မည်သို့ ရွှေ့ပြောင်းသည်ကို ပြောင်းလဲသည့် infrastructure ပြန်လည်တည်ဆောက်မှု ဖြစ်ပြီး ၎င်းတို့ ကိုင်ဆောင်ထားသော tier ကို ပြောင်းလဲခြင်း မဟုတ်ပါ။ မည်သည့်အရာကို upload လုပ်နိုင်သည်ဆိုသည့် Data Security စည်းမျဉ်းများသည် ယခင်အတိုင်းပင် ဆက်လက် သက်ရောက်နေဆဲဖြစ်သည်။
