New in SAM · Feature Guide

Context Management Enhancements

SAM's chatbot now tracks how much of its context window a conversation is using, trims and summarizes history before it overflows, and shows you a real-time budget indicator — so long research conversations degrade gracefully instead of failing outright.

What's New

  • Context window management: a new budget tracking system monitors conversation size and manages it before limits are hit.
  • History summarization: long conversations are summarized in batches rather than simply cut off.
  • Model and configuration updates: default chat model upgraded, plus a new reasoning-effort setting.
  • Chart generation improvements: better validation and clearer tool-calling behavior for chart requests.
  • UI updates: a real-time context budget indicator and a clearer "compacting" state while history is summarized.

Context Window Management

Every chat has a finite context window — the amount of conversation and document content a model can hold at once. This update adds active management of that budget instead of letting a long conversation simply fail when it runs out:

Real-time budget indicator

The chat UI now shows how much of the context window a conversation is using, so you can see it coming rather than being surprised by a failure.

Dynamic trimming & prioritization

Message history and injected documents are trimmed and prioritized intelligently as a conversation approaches its budget, rather than dropped arbitrarily.

Batched history summarization

Once a conversation crosses a length threshold, older messages are summarized in batches — condensed, not deleted — and a "compacting" indicator shows this is happening.

Graceful overflow handling

If a conversation still exceeds the limit, you now get a clear, actionable message instead of a silent or confusing failure.

Chart Generation Improvements

Chart requests now go through explicit validation before generation — better handling of data series and categorical axes, and clearer instructions guiding the model's tool calls, meaning fewer malformed or unexpected charts.

Breaking Change: Company Financials Batch Limit

Reduced from 5 to 3 companies per request. This affects tool-calling behavior in the chatbot, any external integrations using this endpoint, and the performance characteristics of financial data queries. If you have workflows that assumed a 5-company batch, they now need to request in smaller groups.

This is separate from the scorecard batch increase (3 → 50 companies) covered on the Scorecard Rankings page — that change applies to scorecard retrieval specifically, while this change applies to the company financials tool. The two moved in opposite directions for different reasons; check which tool you're calling before assuming a batch size.

Reliability

Extensively tested. The update ships with 1,482 lines of context budget tests, 358 lines of context error-handling tests, and 219 lines of history summarizer tests, alongside enhanced validation coverage for the updated tools. A database migration is included; no manual data migration is required.

Data Handling Still Applies

Automatic summarization and trimming manage the size of a conversation, not its sensitivity. A summarized version of a conversation can still contain Tier 2 or Tier 3 content if the original did.

Summarization doesn't declassify. Don't rely on history compaction to "clean up" a conversation that drifted into sensitive territory. Anonymize before you type, the same as always — see Data Security. If Tier 3 content already reached the wrong tool, that's a same-day report to your project lead regardless of what the chat UI does with it afterward.

การปรับปรุงการจัดการบริบท (Context)

แชทบอทของ SAM ตอนนี้ติดตามว่าการสนทนาใช้ context window ไปมากเท่าใด ตัดทอนและสรุปประวัติการสนทนาก่อนที่จะล้น และแสดงตัวบ่งชี้งบประมาณแบบเรียลไทม์ — เพื่อให้บทสนทนางานวิจัยที่ยาวเสื่อมสภาพลงอย่างนุ่มนวลแทนที่จะล้มเหลวไปเลย

มีอะไรใหม่

  • การจัดการ context window: ระบบติดตามงบประมาณใหม่ตรวจสอบขนาดของบทสนทนาและจัดการก่อนที่จะถึงขีดจำกัด
  • การสรุปประวัติการสนทนา: บทสนทนายาว ๆ จะถูกสรุปเป็นชุด (batch) แทนที่จะถูกตัดทิ้งไปเฉย ๆ
  • การอัปเดตโมเดลและการตั้งค่า: อัปเกรดโมเดลแชทเริ่มต้น พร้อมการตั้งค่า reasoning-effort ใหม่
  • การปรับปรุงการสร้างกราฟ: การตรวจสอบความถูกต้องที่ดีขึ้นและพฤติกรรมการเรียกใช้เครื่องมือที่ชัดเจนขึ้นสำหรับคำขอสร้างกราฟ
  • การอัปเดต UI: ตัวบ่งชี้งบประมาณบริบทแบบเรียลไทม์ และสถานะ "กำลังบีบอัด (compacting)" ที่ชัดเจนขึ้นขณะสรุปประวัติการสนทนา

การจัดการ Context Window

การแชททุกครั้งมี context window ที่จำกัด — ปริมาณเนื้อหาการสนทนาและเอกสารที่โมเดลสามารถเก็บไว้ได้ในคราวเดียว การอัปเดตนี้เพิ่มการจัดการงบประมาณนั้นอย่างเชิงรุก แทนที่จะปล่อยให้บทสนทนายาว ๆ ล้มเหลวไปเฉย ๆ เมื่อใช้งบประมาณหมด:

ตัวบ่งชี้งบประมาณแบบเรียลไทม์

ตอนนี้ UI ของแชทแสดงให้เห็นว่าบทสนทนาใช้ context window ไปมากเท่าใด ทำให้คุณเห็นสัญญาณล่วงหน้าแทนที่จะประหลาดใจกับความล้มเหลว

การตัดทอนและจัดลำดับความสำคัญแบบไดนามิก

ประวัติข้อความและเอกสารที่แทรกเข้ามาจะถูกตัดทอนและจัดลำดับความสำคัญอย่างชาญฉลาดเมื่อบทสนทนาใกล้ถึงงบประมาณ แทนที่จะถูกทิ้งแบบสุ่ม

การสรุปประวัติแบบเป็นชุด

เมื่อบทสนทนายาวเกินเกณฑ์ที่กำหนด ข้อความเก่า ๆ จะถูกสรุปเป็นชุด — ย่อให้กระชับ ไม่ใช่ลบทิ้ง — และตัวบ่งชี้ "กำลังบีบอัด" จะแสดงว่ากำลังเกิดขึ้น

การจัดการการล้นอย่างนุ่มนวล

หากบทสนทนายังคงเกินขีดจำกัด ตอนนี้คุณจะได้รับข้อความที่ชัดเจนและนำไปปฏิบัติได้ แทนที่จะเป็นความล้มเหลวแบบเงียบหรือสับสน

การปรับปรุงการสร้างกราฟ

คำขอสร้างกราฟตอนนี้ผ่านการตรวจสอบความถูกต้องอย่างชัดเจนก่อนสร้าง — การจัดการชุดข้อมูล (data series) และแกนหมวดหมู่ที่ดีขึ้น และคำสั่งที่ชัดเจนขึ้นในการนำทางการเรียกใช้เครื่องมือของโมเดล ทำให้กราฟที่ผิดรูปหรือไม่คาดคิดลดน้อยลง

การเปลี่ยนแปลงที่กระทบระบบเดิม: ขีดจำกัด Batch ของข้อมูลการเงินบริษัท

ลดลงจาก 5 เหลือ 3 บริษัทต่อคำขอ สิ่งนี้ส่งผลต่อพฤติกรรมการเรียกใช้เครื่องมือในแชทบอท การเชื่อมต่อภายนอกใด ๆ ที่ใช้ endpoint นี้ และคุณลักษณะด้านประสิทธิภาพของ query ข้อมูลการเงิน หากคุณมี workflow ที่สมมติว่าเป็น batch 5 บริษัท ตอนนี้จำเป็นต้องขอเป็นกลุ่มที่เล็กลง

สิ่งนี้แยกจากการเพิ่ม batch ของ scorecard (3 → 50 บริษัท) ที่กล่าวถึงในหน้า Scorecard Rankings — การเปลี่ยนแปลงนั้นใช้กับการดึงข้อมูล scorecard โดยเฉพาะ ในขณะที่การเปลี่ยนแปลงนี้ใช้กับเครื่องมือข้อมูลการเงินบริษัท ทั้งสองเปลี่ยนไปในทิศทางตรงข้ามกันด้วยเหตุผลที่ต่างกัน ตรวจสอบว่าคุณกำลังเรียกใช้เครื่องมือใดก่อนที่จะสมมติขนาด batch

ความน่าเชื่อถือ

ทดสอบอย่างละเอียด การอัปเดตนี้มาพร้อมการทดสอบงบประมาณบริบท 1,482 บรรทัด การทดสอบการจัดการข้อผิดพลาดของบริบท 358 บรรทัด และการทดสอบตัวสรุปประวัติ 219 บรรทัด พร้อมด้วยการครอบคลุมการตรวจสอบความถูกต้องที่ปรับปรุงแล้วสำหรับเครื่องมือที่อัปเดต มีการทำ database migration มาให้ ไม่จำเป็นต้องทำการย้ายข้อมูลด้วยตนเอง

หลักการจัดการข้อมูลยังคงใช้บังคับ

การสรุปและตัดทอนอัตโนมัติจัดการ ขนาด ของบทสนทนา ไม่ใช่ ความอ่อนไหว ของมัน บทสนทนาที่ถูกสรุปแล้วยังคงมีเนื้อหา Tier 2 หรือ Tier 3 ได้ หากต้นฉบับมีอยู่แล้ว

การสรุปไม่ได้ลดระดับชั้นความลับ อย่าพึ่งพาการบีบอัดประวัติเพื่อ "ทำความสะอาด" บทสนทนาที่หลงเข้าไปในพื้นที่อ่อนไหว ให้ anonymize ก่อนพิมพ์เสมอเช่นเดิม — ดู ความปลอดภัยข้อมูล หากเนื้อหา Tier 3 ได้ไปถึงเครื่องมือที่ผิดแล้ว นั่นคือรายงานที่ต้องแจ้งหัวหน้าโครงการภายในวันเดียวกัน ไม่ว่า UI ของแชทจะจัดการกับมันอย่างไรในภายหลัง

SAM ရှိ အသစ် · Feature Guide

Context စီမံခန့်ခွဲမှု မြှင့်တင်မှုများ

SAM ၏ chatbot သည် ယခုအခါ စကားဝိုင်းတစ်ခုက context window ကို မည်မျှသုံးနေသည်ကို ခြေရာခံနေပြီး၊ history ကို လျှံမတက်မီ ဖြတ်တောက်ပြီး အကျဉ်းချုပ်ကာ real-time budget indicator ကို ပြသပေးသည် — ထို့ကြောင့် သုတေသန စကားဝိုင်းရှည်များသည် လုံးဝ ပျက်သွားမည့်အစား ညင်သာစွာ အနိမ့်ကျသွားမည်ဖြစ်သည်။

အသစ်များ

  • Context window စီမံခန့်ခွဲမှု: budget tracking system အသစ်တစ်ခုက စကားဝိုင်းအရွယ်အစားကို စောင့်ကြည့်ပြီး ကန့်သတ်ချက်မမီမီ စီမံပေးသည်။
  • History အကျဉ်းချုပ်ခြင်း: ရှည်လျားသော စကားဝိုင်းများကို ဖြတ်တောက်ရုံမျှသာမက batch အလိုက် အကျဉ်းချုပ်ပေးသည်။
  • Model နှင့် configuration update များ: default chat model ကို အဆင့်မြှင့်တင်ပြီး reasoning-effort setting အသစ်တစ်ခု ထပ်ထည့်ထားသည်။
  • ဂရပ်ဖ်ဖန်တီးမှု တိုးတက်မှု: ဂရပ်ဖ် တောင်းဆိုချက်များအတွက် ပိုမိုကောင်းမွန်သော validation နှင့် ပိုမိုရှင်းလင်းသော tool-calling အပြုအမူ။
  • UI update များ: real-time context budget indicator နှင့် history ကို အကျဉ်းချုပ်နေစဉ် ပိုမိုရှင်းလင်းသော "compacting" အခြေအနေ။

Context Window စီမံခန့်ခွဲမှု

chat တိုင်းတွင် ကန့်သတ်ထားသော context window ရှိသည် — model တစ်ခုက တစ်ပြိုင်နက် သိမ်းဆည်းနိုင်သည့် စကားဝိုင်းနှင့် စာရွက်စာတမ်း content ပမာဏ။ ဤ update က စကားဝိုင်းရှည်တစ်ခုကို ချွတ်ယွင်းသောအခါ ရိုးရိုးရှင်းရှင်း ပျက်စေမည့်အစား ထို budget ကို တက်ကြွစွာ စီမံခန့်ခွဲပေးသည်။

Real-time budget indicator

chat UI သည် ယခုအခါ စကားဝိုင်းတစ်ခုက context window ကို မည်မျှသုံးနေသည်ကို ပြသပေးသဖြင့် အံ့ဩစရာ error တစ်ခု ခံစားရမည့်အစား ကြိုတင်မြင်နိုင်သည်။

Dynamic trimming & prioritization

Message history နှင့် ထည့်သွင်းထားသော document များကို စကားဝိုင်းသည် budget အနီးသို့ ရောက်လာသောအခါ ကျိုးကြောင်းဆီလျော်စွာ ဖြတ်တောက်ပြီး ဦးစားပေး သတ်မှတ်ပေးသည်၊ ကျပန်း ဖျက်ချခြင်းမပြုပါ။

Batched history summarization

စကားဝိုင်းသည် အရှည် threshold တစ်ခု ကျော်လွန်သွားသောအခါ အဟောင်း message များကို batch အလိုက် အကျဉ်းချုပ်ပေးသည် — ချုံ့ရုံသာ၊ ဖျက်ခြင်းမဟုတ်ပါ — ပြီးလျှင် "compacting" indicator က ၎င်းဖြစ်ပျက်နေကြောင်း ပြသပေးသည်။

Graceful overflow handling

စကားဝိုင်းသည် ကန့်သတ်ချက်ကို ကျော်လွန်နေဆဲဖြစ်လျှင် ယခုအခါ ရှင်းလင်းပြီး အရေးယူနိုင်သည့် message တစ်ခု ရရှိမည်ဖြစ်ပြီး တိတ်ဆိတ် (သို့) ရှုပ်ထွေးသော error အဖြစ်မှ ကင်းဝေးမည်။

ဂရပ်ဖ်ဖန်တီးမှု တိုးတက်မှု

ဂရပ်ဖ် တောင်းဆိုချက်များသည် ယခုအခါ ဖန်တီးမီ ရှင်းလင်းသော validation ကို ဖြတ်သန်းသည် — data series နှင့် categorical axis များကို ပိုမိုကောင်းမွန်စွာ ကိုင်တွယ်ခြင်း၊ model ၏ tool call များကို ပိုမိုရှင်းလင်းစွာ လမ်းညွှန်ခြင်း — ဆိုလိုသည်မှာ ချို့ယွင်းသော (သို့) မမျှော်လင့်ထားသော ဂရပ်ဖ်များ လျော့နည်းသွားသည်။

ပြောင်းလဲမှု ကြီးမား: ကုမ္ပဏီဘဏ္ဍာရေးဒေတာ Batch ကန့်သတ်ချက်

တောင်းဆိုချက်တစ်ခုလျှင် ကုမ္ပဏီ ၅ ခုမှ ၃ ခုအထိ လျှော့ချထားသည်။ ဤအရာသည် chatbot ၏ tool-calling အပြုအမူ၊ ဤ endpoint ကို အသုံးပြုသည့် ပြင်ပ integration များ၊ နှင့် ဘဏ္ဍာရေးဒေတာ query များ၏ စွမ်းဆောင်ရည် သွင်ပြင်လက္ခဏာများကို ထိခိုက်စေသည်။ ကုမ္ပဏီ ၅ ခု batch ကို ယူဆထားသော workflow များရှိလျှင် ယခုအခါ ပိုသေးငယ်သော အုပ်စုများဖြင့် တောင်းဆိုရန် လိုအပ်မည်။

ဤအရာသည် Scorecard Rankings စာမျက်နှာတွင် ဖော်ပြထားသော scorecard batch တိုးမြှင့်မှု (၃ → ၅၀ ကုမ္ပဏီ) နှင့် သီးခြားစီဖြစ်သည် — ထိုပြောင်းလဲမှုသည် scorecard ရယူခြင်းအတွက်သာ သက်ဆိုင်ပြီး ဤပြောင်းလဲမှုမှာ company financials tool အတွက် သက်ဆိုင်သည်။ ၎င်းတို့နှစ်ခုစလုံးသည် မတူညီသော အကြောင်းရင်းများကြောင့် ဆန့်ကျင်ဘက်လမ်းကြောင်းများသို့ ရွေ့လျားခဲ့သည် — batch size တစ်ခုကို ယူဆမီ မည်သည့် tool ကို ခေါ်နေသည်ကို စစ်ဆေးပါ။

ယုံကြည်စိတ်ချရမှု

အကျယ်တဝင့် စမ်းသပ်ထားသည်။ ဤ update တွင် context budget test line ၁,၄၈၂ လိုင်း၊ context error-handling test line ၃၅၈ လိုင်း၊ history summarizer test line ၂၁၉ လိုင်း ပါဝင်ပြီး update လုပ်ထားသော tool များအတွက် validation coverage ကိုလည်း မြှင့်တင်ထားသည်။ Database migration တစ်ခု ပါဝင်ပြီး ကိုယ်တိုင် ဒေတာပြောင်းရွှေ့ရန် မလိုအပ်ပါ။

ဒေတာကိုင်တွယ်မှု စည်းမျဉ်း ဆက်လက်သက်ရောက်

အလိုအလျောက် အကျဉ်းချုပ်ခြင်းနှင့် ဖြတ်တောက်ခြင်းသည် စကားဝိုင်းတစ်ခု၏ အရွယ်အစား ကို စီမံပေးခြင်းဖြစ်ပြီး ၎င်း၏ အာရုံခံနိုင်စွမ်း ကို မဟုတ်ပါ။ မူရင်းစကားဝိုင်းတွင် Tier 2 (သို့) Tier 3 content ရှိခဲ့လျှင် အကျဉ်းချုပ်ထားသော version တွင်လည်း ဆက်ရှိနိုင်သည်။

အကျဉ်းချုပ်ခြင်းသည် tier ကို မလျှော့ချပေးပါ။ အာရုံခံနိုင်စွမ်းရှိသော နယ်ပယ်ထဲ ရောက်သွားသည့် စကားဝိုင်းတစ်ခုကို "သန့်ရှင်းအောင်" ပြုလုပ်ရန် history compaction ကို မမှီခိုပါနှင့်။ အမြဲတမ်းအတိုင်း စာရိုက်မီ anonymize ပြုလုပ်ပါ — ဒေတာလုံခြုံရေး ကို ကြည့်ပါ။ Tier 3 content သည် မှားယွင်းသော tool ကို ရောက်ရှိပြီးသားဖြစ်ပါက chat UI က နောက်ပိုင်းတွင် မည်သို့ ကိုင်တွယ်သည်ဆိုသည်နှင့် မသက်ဆိုင်ဘဲ project lead ကို ထိုနေ့တွင်ပင် အစီရင်ခံရမည်။
